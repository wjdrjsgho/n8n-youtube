## **AI를 활용한 디자인-코드 핸드오프의 현 문제점**

- 디자인을 코드로 전환하는 과정은 여전히 가장 문제 많은 부분 중 하나임.

- 이상적으로는 디자인 변경 시 코드도 함께 업데이트되는 깔끔한 방법이 있을 것이라 생각함.

- 하지만 현재 사용 가능한 대부분의 툴은 디자인 기능만 제공하거나 코드 기능만 제공함.

- <strong>기존 툴들의 한계점:</strong>

- Stitch 및 Bolt: 프롬프트-투-코드(prompt-to-code) 방식이라 되돌아가 편집할 디자인 캔버스가 없음.

- Figma MCP: 읽기 전용이라 AI가 디자인을 가져올 수는 있어도 디자인할 수는 없음.

- 코딩 에이전트 직접 사용: 디자인 변경 시마다 처음부터 다시 프롬프트를 입력해야 함.

## **Pencil.dev의 등장과 초기 관심**

- Pencil.dev는 디자인과 코드 사이의 양방향 브릿지를 표방하며 주목받기 시작함.

- 최근 디자이너들 사이에서 인기를 얻는 새로운 AI 디자인 툴임.

- <strong>Pencil.dev의 역할:</strong>

- 디자인에 특화된 인기 디자인 툴과 디자인 패턴 구현에 맞춤화된 AI 개발 툴을 연결하는 브릿지 역할.

- 모든 인기 에이전트 ID와 연결됨.

- 컴포넌트 생성, UI 라이브러리 지원, 자동 CSS 클래스 생성 등 디자인-코드 워크플로우를 원활하게 만드는 많은 기능을 포함함.

- 인터페이스: Figma와 가장 유사한 인터페이스를 가짐.

- 초기 사용: 무료로 제공되어 팀에서 테스트를 위해 다운로드함.

## **Pencil.dev 설정 및 핵심 기능**

- <strong>설치 및 연결:</strong>

- 데스크톱 앱 다운로드, 설치된 모든 AI 플랫폼에 연결하는 옵션을 제공함.

- 주력 툴인 Claude Code에 연결함.

- Pencil.dev는 내부적으로 Claude Code 및 Codeex와 같은 AI 코딩 플랫폼을 사용하여 작동함.

- MCP 구성: 데스크톱 앱 설치 후 Pencil.dev용으로 MCP(Managed Code Project)가 자동 구성되며, Claude Code에 모든 툴이 즉시 표시됨.

- 모델 선택: 설정 완료 후 OpenAI 및 Anthropic의 모든 모델을 채팅에서 사용 가능하며, Opus 4.6 모델을 선택함.

- <strong>디자인 관리 기능:</strong>

- 디자인 파일을 `.pen` 확장자의 JSON 기반 형식으로 프로젝트 폴더 내에 저장함.

- 이 파일은 Git을 사용하여 버전 관리가 가능함.

- <strong>가장 큰 특징:</strong> 앱 내의 코딩 에이전트와 디자인 사이의 양방향 브릿지 역할.

## **양방향 브릿지의 실제 작동 방식과 기대의 차이**

- <strong>초기 기대:</strong> Claude 또는 다른 AI 에이전트가 디자인을 코드에, 코드를 디자인에 자동으로 동기화할 것이라 예상함.

- <strong>실제 작동:</strong> 자동 동기화 기능은 없음, 수동으로 프롬프트를 입력해야만 동기화가 이루어짐.

- <strong>동기화 과정:</strong> 수동 프롬프트 입력 시 디자인 섹션을 분석하여 HTML 파일에 구현함. 캔버스에 있는 디자인과 정확히 일치하게 구현됨.

- <strong>문제점:</strong> 디자인 수정 후 매번 이 과정을 반복하는 것이 과도한 오버헤드라고 판단함.

## **자동 동기화 스크립트 개발**

- <strong>개발 배경:</strong> Claude Code와 연결된 Pencil.dev를 사용하여 크리에이터 디렉션 스튜디오의 랜딩 페이지를 작업 중, 디자인 수정이 잦고 반복적인 동기화 작업이 번거로웠음.

- <strong>해결책:</strong> 이 문제를 해결하기 위한 스크립트를 개발함.

- <strong>스크립트 구현 기술:</strong>

- 파일 변경을 모니터링하는 여러 JavaScript 라이브러리를 사용함.

- Claude가 반복적으로 호출되어 토큰을 낭비하고 세션 제한에 도달하는 것을 방지하기 위해 쿨다운 기간을 설정함.

- <strong>스크립트 작동 방식:</strong>

- 디자인이 포함된 `.pen` 파일을 지속적으로 감시함.

- 파일 변경이 감지되면 프로젝트와 동기화하는 방법을 포함하는 프롬프트와 함께 Claude CLI를 실행함.

- 쿨다운 기간 덕분에 각 저장 사이에 간격이 생겨 모든 작은 변경 사항이 한 번에 직접 동기화되지 않음.

- <strong>사용 방법:</strong>

- `npm run sync` 명령을 실행하기만 하면 워커가 즉시 시작되어 디자인 파일인 `.pen` 파일을 모니터링함.

- 이 스크립트가 실행되는 동안 디자인을 수정하고 파일 저장 키 조합을 누르면, 명시적으로 지시하지 않아도 Claude CLI가 자동으로 트리거되어 변경 사항을 프로젝트에 동기화하기 시작함.

- <strong>선행 단계:</strong>

- 스크립트 사용 전, `cloud` 폴더 내 `setting.json` 파일에 읽기, 쓰기, MCP 툴 호출 등 편집에 필요한 모든 권한을 사전 구성해야 함.

- 이를 설정하지 않으면 Claude가 권한 프롬프트에서 무한히 블록되어 프로젝트 변경 및 구현을 제대로 수행할 수 없음.

- <strong>결과:</strong> 이 스크립트 실행으로 더 이상 수동으로 프로젝트 동기화를 프롬프트할 필요가 없어짐. 각 이터레이션은 스크립트에 의해 모니터링되고 자동으로 Claude로 전송되어 구현됨.

- <strong>접근성:</strong> 이 스크립트와 앱의 소스 코드는 AIAS Pro 커뮤니티에서 이용 가능함.

## **멀티 에이전트 시스템을 활용한 확장 구현**

- <strong>아이디어:</strong> Claude Code의 멀티 에이전트 시스템을 Pencil.dev와 함께 사용하는 것을 시도함 (Pencil.dev가 MCP로 연결되어 있기 때문).

- <strong>목표:</strong> 다음 구현을 위해 여러 에이전트를 사용, 각 에이전트가 앱의 다른 측면을 동시에 작업하도록 함.

- <strong>장점:</strong> Claude Code는 PRD(Product Requirements Document) 및 UI 가이드와 같이 컨텍스트로 저장된 모든 문서 파일에 접근할 수 있어 작업이 용이함.

- <strong>적용 사례:</strong> 랜딩 페이지 외에 웹사이트의 다른 페이지들도 구현해야 했음.

- <strong>지시:</strong> 명시적으로 각 에이전트가 구축하려는 앱의 다른 측면을 맡도록 지시함.

- <strong>구현 과정:</strong>

- 탐색을 시작하고 여러 MCP 툴을 호출함.

- 5개 페이지를 위해 Claude는 5개의 에이전트를 생성하고, 각 에이전트가 전담 페이지를 작업하게 함.

- 잠시 후, 애플리케이션 디자인의 초기 버전이 생성됨.

- 앱에 필요한 모든 페이지를 생성하고, 랜딩 페이지와 동일한 폰트와 스타일을 사용하여 디자인이 일치하는지 확인함.

- 디자인이 완료된 후 `Command + S`를 눌러 파일을 저장하자, 스크립트가 작동하여 디자인을 앱에 자동으로 구현하여 수동 프로세스를 가속화함.

## **GSAP을 이용한 스크롤 애니메이션 구현**

- <strong>문제점:</strong> 웹사이트는 견고해 보였으나 페이지에 움직임이 없고 정적인 레이아웃이었음. 시선을 유도할 스크롤 애니메이션이 필요했음.

- <strong>해결책:</strong> GSAP(GreenSock Animation Platform) 라이브러리를 사용함. GSAP은 견고한 JavaScript 애니메이션을 제공하며 복잡한 애니메이션 구현이 용이함.

- <strong>프롬프트 작성:</strong>

- XML 형식으로 매우 상세한 프롬프트를 작성함.

- 이유: Claude 모델은 XML 구조의 프롬프트에서 더 잘 작동하도록 튜닝되어 지시 사항을 더 쉽게 파싱할 수 있음.

- 내용: 작업 세부 사항, 필요한 모든 의존성, 각 특정 요소를 타겟팅하고 각 섹션이 어떻게 동작해야 하는지 정확히 명시하는 중요한 노트들을 해당 태그 내에 포함함.

- <strong>결과:</strong> Claude Code에 이 프롬프트를 제공하자, GSAP 모션이 포함된 컴포넌트들을 추가하고 GSAP 라이브러리를 설치함.

- <strong>웹사이트 변화:</strong> 애니메이션 추가 후 웹사이트는 더 생동감 있고 상호 작용적임. 정적이었던 이전 버전과 매우 다른 사용자 경험을 제공함.

- <strong>프롬프트 접근성:</strong> 이 프롬프트는 AIABs Pro 커뮤니티에서도 이용 가능함.

## **Lennis를 통한 스크롤 경험 향상**

- <strong>추가 구현:</strong> 초기 GSAP 스크롤링 위에 Lennis라는 또 다른 레이어를 추가함.

- <strong>Lennis 설명:</strong> 오픈소스 스무스 스크롤 라이브러리로, 웹사이트에 더 몰입감 있는 레이아웃을 만드는 데 가장 인기 있는 툴 중 하나임.

- <strong>GSAP과의 시너지:</strong>

- GSAP으로 이미 스크롤 애니메이션을 구성했지만 Lennis를 추가한 것은 GSAP과 Lennis가 서로 보완하기 때문임.

- GSAP은 스크롤 시 '무엇이' 일어날지를 제어하고, Lennis는 스크롤 '자체'의 모양과 느낌을 제어함.

- Lennis 없이 스크롤은 위치 간에 점프하지만, Lennis를 사용하면 페이지가 부드럽게 흐르고 GSAP 애니메이션이 트리거될 때 더 자연스럽게 느껴짐.

- <strong>프롬프트 작성:</strong>

- 이 작업을 위해 또 다른 상세한 프롬프트를 작성함.

- GSAP과 동일하게 XML 프롬프트 방식을 따름.

- 내용: Lennis 프롬프트는 이전 단계에서 추가된 기존 스크롤 애니메이션을 향상하는 데 중점을 둠. 의존성, 모든 설정 지침, 요소에 대한 상세 설명, 애니메이션 배치 방법, 동작 방식, 그리고 규칙을 포함함.

- <strong>결과:</strong> 이 프롬프트를 Claude Code에 제공하자, 리포지토리 전체에 걸쳐 많은 변경 사항을 적용함. 모든 구현이 완료된 후 웹사이트를 확인한 결과, 현저히 개선되었음. 가장 큰 변화는 더 부드러운 스크롤링이 추가되어 웹사이트 탐색이 훨씬 더 몰입적인 경험이 된 것임.

## **UX 감사 스킬을 통한 UX 표준 준수 확인**

- <strong>최종 단계:</strong> 모든 기능이 구축되고 준비된 웹사이트의 UX 표준 준수 여부를 확인하는 것이 마지막 단계임.

- <strong>스킬 생성:</strong> 스킬 크리에이터를 사용하여 'UX 감사(UX audit)'라는 스킬을 생성함.

- <strong>스킬 목표:</strong> UI 요소의 품질을 확인하고, 웹사이트의 다양한 측면을 검토하며, 모든 것이 UX 표준을 준수하는지 확인하는 데 중점을 둠.

- <strong>스킬 구성:</strong>

- 여러 단계: 컨텍스트 수집, 9가지 항목의 체크리스트 분석, 그리고 보고 단계(일련의 기준에 따라 점수를 매김).

- 참조 자료: 9가지 항목의 세부 정보가 담긴 UX 체크리스트와, 사람의 눈으로는 놓칠 수 있는 UX 문제점을 프로그램적으로 찾아내기 위한 Python 스크립트가 포함됨.

- <strong>감사 실행 및 결과:</strong>

- 감사 스킬을 실행하자 모든 문제점을 취합하고 두 가지 치명적인(critical) 문제와 여러 주요(major) 및 사소(minor)한 문제를 플래그함.

- 치명적인 문제 중 하나는 색상 대비였음.

- 식별된 모든 문제로 인해 웹사이트는 C 등급을 받음.

- 필요한 모든 변경 사항을 상세히 제공하고, 이러한 수정 사항을 구현하도록 요청함.

- <strong>수정 후 재감사:</strong>

- 모든 변경 사항이 적용된 후 웹사이트는 외관상 크게 달라 보이지 않았지만, 사용성 면에서 크게 향상되어 탐색이 더 쉬워지고 WCAG(Web Content Accessibility Guidelines)를 준수하게 됨.

- 감사 스킬을 다시 실행하자, 웹사이트 등급이 C에서 B로 향상되었음. 이는 모든 주요 문제가 해결되었고 소수의 사소한 문제만 남았음을 의미함.
