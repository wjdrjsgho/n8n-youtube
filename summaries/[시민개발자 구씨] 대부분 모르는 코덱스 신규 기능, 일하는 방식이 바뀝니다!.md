## **코덱스 쓰레드 관리 기능의 필요성 및 해결책**

- 기존 코덱스 사용 시 겪는 문제점 언급함.

- 단일 쓰레드에서 리서치, 기획, 검토 등 여러 업무 진행 시 시간 오래 걸려 답답함을 느낌.

- 병렬로 여러 쓰레드 나눠 작업 시 쓰레드 관리가 어려움을 겪음.

- 코덱스 최신 업데이트가 해당 문제 해결함을 알림.

- 코덱스 쓰레드 내에서 프롬프트로 쓰레드 관리가 가능해짐.

- 업데이트된 기능을 활용하여 여러 쓰레드로 작업을 나누고 효율적으로 관리하는 시스템을 소개함.

- 혼자 또는 적은 인원으로 리서치, 콘텐츠 제작, 영업 관리 등 다양한 업무를 해야 하는 경우 특히 도움될 것임을 언급함.

## **코덱스 활용 AI 조직 비유 및 워크플로 구조**

- 코덱스를 단순한 코딩 도구가 아닌 AI 조직으로 생각할 것을 제안함.

- 핵심은 부서를 나눠 작업을 시키는 것임.

- 워크플로 구조 설명함.

- <strong>팀장 방 (메인 쓰레드):</strong>

- 직접 업무 처리하지 않음.

- 어떤 방에 무슨 업무를 시킬지 정하고 지시함.

- 각 방의 내용을 보고받아 최종적으로 화자에게 보고함.

- <strong>담당자 방 (하위 쓰레드):</strong>

- 실제로 업무를 진행하는 역할임.

- 예시: 리서치 방 (자료 찾고 정리), 글 쓰기 방 (자료 기반 글 작성), 이미지 제작 방 (특정 이미지 제작).

- 각 방은 회사의 팀 단위로 나눠서 업무를 하는 것과 유사하다고 설명함.

- <strong>서브 에이전트 활용:</strong>

- 필요에 따라 방으로 나뉘는 것 외에, 나눠진 쓰레드 안에서 서브 에이전트도 호출하여 작업을 더 쪼개서 진행하고 결과를 종합할 수 있음.

- <strong>보고 및 피드백 흐름:</strong>

- 각 방에서 업무 완료 시 팀장 방으로 보고함.

- 팀장이 보고받아 화자에게 최종 컨펌을 받음.

- 화자의 피드백이 있으면 팀장이 각 방에 필요한 피드백을 제공하고 추가 작업을 요청함.

## **쓰레드를 나눠 작업하는 방식의 가치 평가 기준**

- 단순하고 짧은 일은 업무방 하나에서 진행하는 것이 가장 빠르고 단순함을 인정함.

- 쓰레드를 나누는 방식의 가치는 다음 세 가지 기준으로 평가할 수 있음을 제시함.

- <strong>동시에 업무 진행 가능 여부:</strong>

- 예시: 리서치 작업은 다양한 방면의 리서치를 동시에 진행해도 무리가 없는 경우가 많음.

- 동시 작업 시 최종 결과물을 더 빨리 받아볼 수 있어 병렬 작업이 상관없는 경우 유용함.

- <strong>일의 종류가 다른가:</strong>

- 예시: 리서치 작업, 글쓰기 작업, 대시보드 제작 등은 특성이 다름.

- 회사의 다른 팀이 할 법한 업무의 경우 쓰레드를 나눠서 작업하는 것이 더 효율적일 수 있음.

- <strong>중간 결과를 넘겨야 하는가:</strong>

- 예시: 리서치 결과를 기반으로 글쓰기를 하는 경우처럼 단계별 업무 관리가 필요한 경우.

- 단계별 업무를 관리하는 데 더 효율적임.

- 위 세 가지 중 하나라도 해당된다면 쓰레드를 나눠서 작업하는 것을 고려해 볼 만함.

## **뉴스레터 제작 시나리오를 통한 실제 워크플로 시연**

- 코덱스 앱을 다운로드하여 실행하고 로컬 프로젝트를 생성함.

- 프로젝트 이름을 '뉴스레터'로 설정함.

- 시나리오: 비개발자를 위한 AI 트렌드 뉴스레터 제작.

- 뉴스레터 업무는 자료 조사, 팩트 체크, 목차 작성, 글쓰기, 이미지 삽입, HTML 형식 변환 등 여러 단계를 거치고 특성이 명확히 나뉘어 해당 구조에 적합함을 설명함.

- <strong>팀장 방 (메인 쓰레드) 설정:</strong>

- 새로운 챗에 프롬프트를 넣어 지시함.

- 해당 쓰레드가 프로젝트의 팀장 방임을 명시함.

- 팀장 방은 직접 리서치나 글을 쓰지 않고, 담당자 방에 일을 맡기고 지시, 보고 취합, 다음 단계 진행 결정만 담당하도록 함.

- 프로젝트 목표는 '비개발자를 위한 AI 트렌드 뉴스레터 제작'으로 설정함.

- 담당자 방(쓰레드) 구성 및 진행 순서, 게이트 설정을 상세히 지정함.

- 예시: 리서치 병렬 진행 -> 확인 -> 팩트 체크 -> 아웃라인 작업 -> 라이팅 작업 등.

- 보고 규칙 설정: 각 방에서 결론을 먼저 제시하고 근거를 나중에 보고하도록 지시함.

- 별도 쓰레드에서 작업 완료 시, 팀장 방에서 확인하고 완료된 경우 항상 알아서 보고하도록 요청함.

- 프롬프트는 가이드 문서로 제공하여 참고할 수 있도록 함.

- <strong>병렬 리서치 작업 진행:</strong>

- '진행해' 지시 후 작업 완료 시 홀딩하여 결과 보고 요청함.

- 뉴스레터 프로젝트 안에 '리서치 뉴스', '리서치 툴', '리서치 케이스' 세 개의 쓰레드가 생성됨.

- 각 쓰레드에서 별도의 프롬프트로 요청이 들어감 (by code from another thread).

- 서브 에이전트보다 세션을 나눠 진행하는 것이 이력 확인에 더 유용함을 설명함.

- 폴링 간격 조절 가능함을 언급함 (기본 30초).

- <strong>리서치 결과 보고 및 피드백:</strong>

- 메인 쓰레드에서 각 리서치 쓰레드의 결과 요약을 보고함.

- 예시: AI 흐름, 관련 툴, 실전 사례 후보 등.

- 추가 피드백 가능함: '툴 쪽에서 AI 에이전트 내용에 대해 더 리서치해 달라'고 요청함.

- 폴링 시 완료되지 않은 내용은 출력하지 말고 최종 완료되었을 때만 보고 내용 출력하도록 요청함.

- 추가 메시지가 '리서치 툴' 쓰레드로만 전달됨을 확인함.

- 추가 리서치 완료 후 메인 방에 보고됨.

- 보고 내용을 구체적으로 확인하려면 우측에서 리포트 내용을 바로 확인 가능함을 언급함.

- <strong>팩트 체크 작업 진행:</strong>

- 리서치 결과 확인 및 보강 리서치 완료 후 다음 단계인 팩트 체크 진행을 요청함.

- '팩트 체크' 쓰레드가 추가로 생성됨.

- 검증 대상 파일을 첨부하고 검증 원칙에 따라 내용 검증 및 팩트 체크 문서 생성을 지시함.

- 화자의 '진행해 줘'라는 간단한 요청에도 알아서 작업을 잘 실행함을 언급함.

- 팩트 체크 완료 후 보고 내용을 우측에서 확인 가능함.

- 추가 팩트 체크 요청도 가능함을 언급함.

- <strong>아웃라인 (목차) 작업 진행:</strong>

- 팩트 체크 통과 및 수정 가능 자료를 바탕으로 뉴스레터 목차와 섹션 구조를 잡는 아웃라인 작업을 요청함.

- 메인 방에서 보고 문서를 확인하고 피드백을 주며 다음 단계 진행을 지시하는 팀장의 역할임을 강조함.

- '아웃라인' 쓰레드가 추가로 생성되어 작업을 진행함.

- 아웃라인 완료 후 결론과 목차 구조를 보고함.

- 특정 목차를 선택하여 이후 단계 진행을 요청함.

- <strong>라이팅 및 이미지 제작 병렬 작업 진행:</strong>

- 선택된 목차 기반으로 라이팅과 이미지 작업을 병렬로 시작하겠다고 알림.

- 이미지 제작은 실제 이미지 생성보다는 어떤 이미지를 생성할지에 대한 구조(프롬프트)를 잡는 작업임을 언급함.

- '라이팅'과 '이미지' 쓰레드가 각각 추가로 생성되어 병렬로 작업을 진행함.

- 각 쓰레드에 별도로 작업 요청이 들어감을 확인함.

- 작업 완료 후 라이팅 쓰레드에서는 글 내용과 검증 출처를, 이미지 쓰레드에서는 이미지 생성 프롬프트를 보고함.

- <strong>최종 뉴스레터 제작:</strong>

- 완성된 라이팅과 이미지 자료를 바탕으로 최종 뉴스레터 제작을 요청함.

- HTML 버전과 PDF 버전을 생성할 것임을 알림.

- 디자인 MD 문서 예시를 참고하여 적용해 달라고 요청함.

- '패키징' 쓰레드에서 작업을 진행하나, 디자인 MD 반영을 누락하는 상황이 발생함.

- 팀장 방(메인 쓰레드)에서 이를 직접 체크하여 하위 쓰레드의 누락 사항을 발견하고 재작업을 요청함. 이는 팀장의 역할을 명확히 보여주는 예시임.

- 최종적으로 HTML 및 PDF 버전의 뉴스레터가 생성됨.

- 생성된 HTML 버전을 통해 디자인 MD (클로드 느낌의 색상)가 반영된 것을 확인함.

## **코덱스 쓰레드 관리 구조의 장점 및 추가 활용 방안**

- <strong>장점:</strong>

- 메인 방에서는 처음 요청부터 최종 결과물까지 깔끔하게 중간 단계별 보고 내용만 한눈에 확인 가능함.

- 세부적인 실무 작업은 별도의 쓰레드로 분리되어 메인 쓰레드에서는 결과물 위주로 확인 및 피드백만 가능하여 운영 및 관리 용이함.

- <strong>추가 활용 방안:</strong>

- <strong>`agent.md` 활용:</strong>

- 매번 복잡한 프롬프트를 줄 필요 없이, 현재 프로세스를 `agent.md` 파일로 작성하여 프로젝트 폴더에 저장함.

- 이후에는 "이번 주 뉴스레터 생성을 해보자"와 같은 간단한 프롬프트만으로도 뉴스레터 작업을 손쉽게 진행할 수 있음.

- <strong>스킬 매칭:</strong>

- 더 높은 퀄리티 컨트롤을 위해 `agent.md`뿐만 아니라 스킬을 만들고 각각의 쓰레드 방과 매칭시킬 수 있음.

- 현재는 코덱스가 알아서 세부 프롬프트를 생성하지만, 특정 작업(예: 이미지 준비, 라이팅)에 대한 세부적인 진행 원칙을 스킬로 정의하여 일관된 결과물을 받아볼 수 있음.

## **핵심 요약 및 주의사항**

- <strong>핵심:</strong> 코덱스에서 복잡한 업무를 쓰레드로 쪼개 진행하고, 메인 쓰레드에서 전체 프로세스를 관리하는 구조는 매우 유용할 수 있음.

- 특히 병렬 작업을 많이 하거나, 직무별로 작업을 나눠 컨텍스트를 분리 관리하고 싶을 때 유용함.

- <strong>주의사항 및 한계:</strong>

- <strong>안정성 문제:</strong> 아직 모든 환경에서 안정적이지 않을 수 있음.

- 예시: 원치 않는 백그라운드 세션 실행, 모바일에서 쓰레드 생성 에러 발생 등.

- 버그는 향후 픽스될 수 있으나, 아직 출시 초기 기능으로 예상치 못한 방식의 쓰레드 관리가 발생할 수 있음을 참고해야 함.

- <strong>불필요한 토큰 발생 가능성:</strong>

- 예시: 팀장 방에서 진행 상황을 지속적으로 폴링하여 확인하는 과정에서 토큰이 발생함.

- 토큰을 최소화하는 방식으로 설정하여 비용을 관리하는 것이 중요함.

- 이는 조직에서 관리자를 채용하는 것과 유사한 비용으로 볼 수 있음.

- 관리자 채용 시 커뮤니케이션 비용은 줄지만, 관리자 비용과 계층 구조 형성의 비용이 발생함.

- 모든 업무에 이 구조를 적용하는 것은 비효율적일 수 있음.

- 업무의 특성이 나뉘고 여러 단계를 매니지해야 하는 상황에서, 추가 토큰 비용이 발생하더라도 커뮤니케이션 코스트 절감 및 깔끔한 작업 내용 관리 등 장점이 더 클 때만 활용하는 것이 좋음.
