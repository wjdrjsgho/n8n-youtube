## **서론: 클로드 코드 활용의 중요성**

<ol>

- 일부 개발자들은 다른 이들보다 훨씬 빠르게 코드를 작성하고 배포함.

- 이러한 차이는 지능의 문제가 아니라, '클로드 코드'를 얼마나 독하고 악랄하게 활용하는가의 문제임을 설명함.

- 클로드 코드의 창시자 보리스 체니는 주 100개의 PR을 혼자 처리하는 괴물로 소개됨.

- 보리스는 잠을 자지 않는 것이 아니라, 15개의 클로드 코드 분신을 활용하여 병렬 처리, 자가 치유 시스템, 집단 지성 등을 구현하고 있음.

- 보리스가 공개한 13가지 셋업은 단순한 코딩 팁을 넘어 모든 개발자가 갖춰야 할 필수 생존 키트임을 강조함.

- 이 영상 하나로 클로드 코드 창시자가 인증한, 남들보다 10배 앞서가는 클로드 코드 사용법을 알려줄 것임.
</ol>

## **클로드 코드 창시자가 제시하는 13가지 활용법**

<ol>

- <strong>병렬 실행 및 세션 활용</strong>
    <ol>

- 보리스는 기본적으로 로컬에서 5개의 클로드를 병렬로 실행함.

- 사용자 입력이 필요할 때는 시스템 알림이 실행되도록 설정하여, 클로드가 입력을 요구할 때 즉시 반응할 수 있게 함.

- 로컬 세션 외에 5~10개의 웹 세션을 추가로 실행하여, 총 15개 정도의 세션을 동시에 활용함.

- 클로드 코드에서 `&`를 실행하면 즉시 백그라운드 태스크로 넘길 수 있으며, 보리스는 대체적으로 이 기능을 사용함.

- 가끔은 웹이나 모바일 앱에서 직접 실행하기도 함.
    </ol>

- <strong>최적 모델 사용 및 작업 모드 설정</strong>
    <ol>

- 항상 오퍼스 4.5(Opus 4.5) 모델을 사용하며, 항시 'think' 모드를 켜두고 사용함.

- 오퍼스 4.5가 다른 모델보다 크고 무겁지만, 초기 출력이 우수하고 사람의 개입이 적어도 되어 결과적으로 더 빠르게 작업을 완료하게 됨.
    </ol>

- <strong>팀 단위의 `claude.md` 파일 관리</strong>
    <ol>

- `claude.md` 파일은 Git에 커밋하도록 설계된 파일임.

- 클로드가 무언가를 잘못했을 때, 다음에는 같은 실수를 하지 않도록 `claude.md`에 추가 지시사항을 저장해 둠.

- 팀원 모두가 `claude.md` 파일을 개선하는 데 매우 적극적이며, 팀별로 각자의 `claude.md` 파일을 관리함.
    </ol>

- <strong>PR 검증 시 클로드 태그 및 GitHub 연동</strong>
    <ol>

- PR을 검증할 때 클로드를 태그하여 `claude.md`에 무언가를 추가함.

- `install` 및 `command`를 사용하면 GitHub에서도 클로드를 사용할 수 있음을 알림.
    </ol>

- <strong>플랜 모드를 통한 체계적인 작업</strong>
    <ol>

- 대부분의 세션을 '플랜 모드'에서 시작함.

- Shift+Tab을 누르면 세션 유형을 변경할 수 있으며, 플랜 모드로 진입하면 '리니어 모드'로 자동 전환됨.

- 계획을 여러 번 수정하여 마음에 드는 플랜이 나왔을 때 '오토 액셉 에딧 모드'로 실행하면 보통 한 번에 원하는 결과물이 나옴.

- 좋은 플랜이 결국 좋은 코드를 만들어낼 수 있다는 점을 강조함.
    </ol>

- <strong>워크플로우 자동화를 위한 커맨드 저장</strong>
    <ol>

- 반복적으로 사용하는 워크플로우는 커맨드로 저장하여 똑같은 프롬프트를 계속 입력하지 않도록 함.

- 클로드가 배시(bash) 커맨드를 사용할 수 있도록 하면 엄청난 시간 절약이 가능함.

- 이 커맨드들은 Git에 같이 커밋하여 팀원들과 공유함.
    </ol>

- <strong>서브에이전트 활용을 통한 전문화</strong>
    <ol>

- 몇몇 서브에이전트들을 규칙적으로 사용함.

- 예를 들어 '코드 심플리파이어(Code Simplifier)' 서브에이전트는 클로드 코드가 작업을 완료했을 때 코드를 정리하는 데 사용함.

- '베리파이(Verify)' 에이전트는 클로드 코드가 엔드-투-엔드(end-to-end) 테스트와 같은 테스트를 잘 수행할 수 있는 자세한 명세가 적혀있음.

- 커스텀 커맨드와 마찬가지로, 커스텀 서브에이전트는 항상 사용하는 기능들을 자동화하는 목적으로 생각함.
    </ol>

- <strong>`post_tool_hook`을 이용한 코드 포매팅</strong>
    <ol>

- `post_tool_hook`을 사용해 클로드가 작성한 코드를 포매팅함.

- 클로드는 보통 포매팅이 매우 잘된 코드를 생성하지만, 이 훅을 사용하면 혹시 모를 10%의 문제를 자동으로 매번 해결할 수 있음.
    </ol>

- <strong>안전한 권한 관리를 위한 설정</strong>
    <ol>

- `dangerously_skip_permission_mode`는 사용하지 않음.

- 대신 `permission` 슬래시 커맨드를 사용하여 사용 가능한 배시(bash) 커맨드들을 미리 등록해두고 지속적인 권한 허가가 필요 없도록 설정함.

- 권한은 `~/.claude/settings.json`에 저장하여 팀과 공유함.
    </ol>

- <strong>클로드에게 광범위한 툴 사용 권한 부여</strong>
    <ol>

- 클로드에게 사용할 모든 툴 권한을 제공함.

- 예를 들어 슬랙(Slack) 툴을 검색하거나, 빅쿼리(BigQuery) 쿼리를 실행하고, 분석 요청에 대답하거나, 센트리(Sentry)에서 에러 로그를 확인하는 것처럼 다양한 툴 사용 권한을 부여함.
    </ol>

- <strong>장기 실행 태스크(Long-Running Task) 효율적 처리</strong>
    <ol>

- 클로드가 완료한 작업은 무조건 백그라운드 에이전트를 사용하여 검증하도록 함.

- `agent_stop_hook`을 사용하여 더욱 결정론적인(deterministic) 결과를 얻음.

- 윌건(Willgun) 플러그인을 사용하여 무한 이터레이션 작업을 수행함.
    </ol>

- <strong>클로드의 자체 작업 검증 체계 구축</strong>
    <ol>

- 클로드 코드에게 자신이 한 작업을 검증할 수 있는 방법을 정확히 제공함.

- 클로드 자체적으로 피드백을 생성하도록 하면 2~3배 이상의 좋은 결과를 낼 수 있음.

- 검증 과정은 도메인별로 모두 방식이 다를 수 있으며, 각 도메인별로 브라우저를 실행해서 테스트하거나 모바일을 시뮬레이션해서 테스트하거나 단순히 배시(bash) 커맨드를 실행하는 등 가장 적합한 방법으로 검증 방식을 제시함.
    </ol>

</ol>

## **결론: 행동의 중요성**

<ol>

- 보리스 체니가 말해준 생존 키트들을 보고 누군가는 감탄만 하고 지나칠 것이며, 누군가는 오늘 당장 터미널을 켜고 `claude.md`를 만들고 커스텀 커맨드 및 커스텀 서브에이전트를 제작할 것임.

- 6개월 뒤 이 두 부류의 사람들의 수익과 기술적 격차는 엄청나게 벌어질 것임.

- 심지어 감탄만 하고 넘겼던 사람이 그때도 과연 개발자로 살아남을 수 있을지 의문임을 제시함.

- 변화를 만들어내는 것은 '액션'이며, 기술은 기다려주지 않으며 특히 지금은 더 그러함.

- 그냥 무지성으로 지금 당장 시작할 것을 강조함.
</ol>
