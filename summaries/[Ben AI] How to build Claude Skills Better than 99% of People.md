## **2026년 AI 에이전트 시대에 스킬 엔지니어링의 중요성**

- 클라우드 코드, 코워크, 코드엑스 등의 AI 에이전트가 나날이 강력해짐에 따라, 2026년에는 뛰어난 스킬을 구축하는 것이 가장 중요한 역량 중 하나가 될 것임.

- 에이전트 스킬 구축은 누구든 프롬프팅만으로 모든 워크플로우를 자동화할 수 있게 하며, 심지어 스스로 개선될 수도 있음.

- 우수한 스킬은 곧 수익화될 가능성이 높음.

- 이 가이드는 클라우드 코드, 코워크, OpenAI, Google 등 스킬을 통합한 모든 플랫폼 작업자에게 도움을 줄 예정임.

## **스킬 학습의 필요성: 기존 방법론과의 차이점**

- AI 에이전트가 매우 강력해지고 있음에도 불구하고, 여전히 특정 가드레일, 컨텍스트, 그리고 비즈니스의 고유한 운영 방식과 소프트웨어 사용법에 대한 SOP(표준 운영 절차)가 필요함.

- <strong>기존 솔루션과 한계점:</strong>

- <strong>프로젝트 및 커스텀 GPTs:</strong>

- 시스템 프롬프트를 설정하고 컨텍스트 파일을 추가하는 방식임.

- 문제점은 일반적으로 독립적이며, 여러 창 사이를 이동해야 하고, 스스로 개선되지 않으며, 많은 컨텍스트를 처리하지 못함.

- <strong>NAND 및 자동화 플랫폼:</strong>

- 가드레일을 하드코딩하여 시스템을 결정론적으로 만듦.

- 사람 개입 없는 완전 결정론적 워크플로우에는 여전히 훌륭함.

- 그러나 대부분의 일상 업무는 결정론적이지 않으며, 프로세스가 항상 명확하지 않고, 컨텍스트 의존적이며, 판단을 요구함.

- 이것이 AI로부터 좋은 결과물을 얻기 위해 사람의 개입(Human in the loop)이 중요한 이유임.

- <strong>스킬의 위치와 장점:</strong>

- 스킬은 위 두 가지 방법론의 중간에 위치함.

- 본질적으로 AI 에이전트에게 특정 프로세스를 수행하는 방법에 대한 지침을 제공함.

- 스스로 개선될 수 있고, 사람의 개입이 가능함.

- 프로젝트나 커스텀 GPT와 달리, 수천 개의 스킬에 하나의 에이전트가 접근할 수 있음.

- 핵심적인 변화는 프롬프팅만으로 스킬을 생성하고 업데이트할 수 있다는 점임.

- 이는 클라우드 코드, 코워크에서 단일 인터페이스를 통해 누구든 자신의 작업과 워크플로우를 자동화하기 시작할 수 있음을 의미함.

- 스킬은 비즈니스 전반에 걸쳐 공유 가능함. 한 사람의 프로세스와 도메인 전문 지식을 전체 팀이 즉시 활용할 수 있어, 온보딩, 일관성, 회사 운영 방식에 큰 영향을 미침.

- 하나의 매우 유능한 일반 에이전트에게 수천 개의 특정 기능을 부여할 수 있다면, AI 에이전트가 서서히 작업을 수행하는 유일한 인터페이스가 될 가능성이 높음.

- 이러한 에이전트가 특정 작업을 잘 수행하도록 스킬 인프라를 구축하는 것은 개인과 회사 생산성을 크게 향상시킬 뿐만 아니라, 새로운 소프트웨어 레이어처럼 수익화될 가능성도 있음.

- 하지만 소프트웨어 엔지니어링, 프롬프트 엔지니어링, AI 자동화 엔지니어링과 마찬가지로, 스킬 구축에 능숙해져야 함.

- 누구든 프롬프트를 통해 스킬을 만들 수 있지만, 동일한 스킬이라도 구축 방식에 따라 완전히 다른 결과물을 생성할 수 있음.

- 예를 들어, 급하게 만든 인포그래픽 스킬과 많은 노력을 기울여 만든 스킬의 결과물은 명확성과 브랜드 정렬 면에서 큰 차이를 보임.

## **에이전트 스킬의 구성 요소**

- 에이전트 스킬은 본질적으로 에이전트가 더 정확하고 효율적으로 작업을 수행하는 데 사용할 수 있는 지침, 스크립트 및 리소스 폴더임.

- <strong>핵심 요소: skill.md 파일</strong>

- 이 파일은 프로세스를 수행하는 방법에 대한 지침 역할을 함. 시스템 프롬프트, 클라우드 프로젝트, 또는 커스텀 GPT와 유사하게 볼 수 있음.

- 그러나 프로세스 지침 외에 다음 사항에 대한 추가 지침을 포함할 수 있음:

- 지식 파일 사용 방법 및 시기

- 도구 사용 방법 및 시기

- 하위 에이전트 시작 방법 및 시기

- 코드 실행 사용 방법 및 시기

- 따라서 skill.md는 SOP(표준 운영 절차)로 볼 수 있음.

- <strong>스킬을 더욱 강력하게 만드는 추가 참조 파일 (컨텍스트):</strong>

- <strong>참조 파일 없음 (가장 간단한 스킬):</strong>

- skill.md 또는 프로세스 지침만으로 구성됨.

- 예: Enthropic의 영업 계정 조사 스킬은 skill.md만 포함하며, 실행 흐름(요청 구문 분석, 웹 검색, 풍부화, CRM 확인)을 명시함.

- <strong>텍스트 파일:</strong>

- 스킬에 더 많은 정보를 제공하기 위해 추가하는 일반적인 파일임.

- 포함할 수 있는 일반적인 예시: 예시 출력, 스타일 가이드, ICP(이상적인 고객 프로필)에 대한 컨텍스트, 배경 정보 등.

- 예: 뉴스레터 작성 스킬에는 화자의 배경, 하는 일, ICP, 목소리 개성에 대한 컨텍스트를 제공하는 여러 참조 파일이 포함됨.

- MCP(Model-Controller-Presenter) 지침 파일도 흔히 사용되며, 특정 스킬 프로세스에서 에이전트가 특정 도구를 효율적으로 사용하는 방법을 설명함 (예: CRM 탐색 또는 사용 방법). 에이전트가 이러한 MCP 문서를 구축할 수 있음.

- <strong>자산 또는 비텍스트 파일:</strong>

- 이미지, 프레젠테이션, 비디오 또는 바이너리 파일 등이 포함될 수 있음.

- 예: 원하는 프레젠테이션 레이아웃의 좋은 출력 예시를 제공할 때 사용함.

- <strong>코드 스크립트:</strong>

- Python 또는 JS 함수가 포함될 수 있으며, 소프트웨어에서 API 호출을 수행하거나 함수를 실행하는 등의 작업을 수행할 수 있음.

- 예: 인포그래픽 스킬에는 Google API 호출을 수행하는 스크립트가 포함되어 있음.

- 이처럼 스킬은 매우 간단할 수도 있고, 소프트웨어와 유사하게 상당히 복잡해질 수도 있음. 스킬은 AI 에이전트를 위한 소프트웨어인 셈임.

## **점진적 공개(Progressive Disclosure)를 통한 스킬 접근**

- 하나의 AI 에이전트가 어떻게 수천 개의 스킬에 접근하고, 모든 문서를 컨텍스트 과부하 없이 공급받을 수 있는지에 대한 설명임.

- 이것은 '점진적 공개'라는 프로세스를 통해 이루어짐.

- 새로운 스킬을 생성하거나 업로드하여 AI 에이전트에게 제공하면, 에이전트 메모리에는 메타데이터(설명과 이름)만 저장됨.

- 예: 인포그래픽 스킬의 이름과 설명이 에이전트에 저장됨.

- 이 메타데이터를 통해 에이전트는 특정 스킬을 언제 트리거하거나 사용해야 할지 인지함.

- 스킬이 트리거될 때만 skill.md(프로세스)가 에이전트의 컨텍스트 창에 로드되어 해당 특정 프로세스를 실행하는 방법을 이해함.

- 스킬이 참조 파일 사용을 지시할 때만 해당 참조 파일이 컨텍스트 창에 로드됨.

- 이러한 컨텍스트의 점진적 공개 덕분에 하나의 에이전트가 수천 개의 잠재적 스킬에 접근할 수 있음.

## **플러그인과 스킬의 차이점**

- 플러그인과 스킬 간의 혼동이 많아 빠르게 설명함.

- 플러그인은 본질적으로 스킬, 명령, 에이전트 및 커넥터의 패키지 또는 번들 세트임.

- <strong>플러그인의 세 가지 주요 특징:</strong>
        <ol>

- <strong>복잡성 계층 추가:</strong>

- 플러그인은 여러 스킬을 포함할 수 있음.

- 여러 스킬을 순차적으로 트리거하여 더 복잡한 작업을 자동화하는 워크플로우 트리거 역할을 하는 명령을 포함할 수 있음.

- 전문 에이전트 팀과 사전 설정된 커넥터 세트를 가질 수 있음. 이를 통해 플러그인에 훨씬 더 많은 기능을 추가할 수 있음.

- <strong>쉬운 공유 및 분할:</strong>

- 플러그인을 통해 이러한 번들 패키지를 회사 부서 간에 쉽게 공유하고 나눌 수 있음.

- 영업팀은 영업 플러그인을, 마케팅팀은 마케팅 플러그인을 가질 수 있으며, 각 부서에 필요한 커넥터와 스킬을 포함함.

- <strong>버전 관리 및 업데이트 가능성:</strong>

- 플러그인(스킬도 마찬가지)은 버전 관리가 가능해지고 있으며, 이는 사용하는 모든 계정에서 언제든지 업데이트할 수 있음을 의미함.

        </ol>

- 이러한 특성 때문에 플러그인은 소프트웨어(SaaS)와 매우 유사하게 보이기 시작함.

- 앞으로 SaaS 기업들이 특정 기능을 가진 자체 플러그인을 출시할 것으로 예측함.

- 하지만 플러그인 내부에서도 스킬이 핵심적인 측면임. 플러그인이 있더라도 플러그인 없이도 스킬을 직접 구축하고, 트리거하며, 접근할 수 있음.

## **스킬 및 플러그인의 계층 구조**

- 스킬과 플러그인은 모든 산업, 비즈니스, 부서 및 틈새 워크플로우에 관련됨.

- 세 가지 계층의 스킬과 플러그인이 나타나고 있음:

- <strong>일반 플러그인 및 스킬:</strong>

- Entropic 또는 OpenAI에서 구축한 것과 같은 일반적인 스킬 및 플러그인.

- 이미 Skills MP Smithy와 같은 스킬 마켓플레이스가 생겨나 사람들이 스킬을 구축하고 판매할 가능성이 있음.

- SaaS 기업들이 자체 플러그인을 만들기 시작할 예정임.

- 이들은 여러 산업 및 비즈니스에 관련될 수 있도록 범용 스킬이어야 함.

- <strong>회사 맞춤형 스킬:</strong>

- 기업들은 고유한 프로세스를 기반으로 자체 스킬을 구축하거나, 일반 스킬을 맞춤 설정하기를 원할 것임.

- 예: Entropic 또는 타사 영업 아웃리치 스킬이 유용할 수 있지만, 회사는 회사 브랜드 톤, ICP, 비즈니스 컨텍스트를 추가하여 이를 맞춤 설정하고 개선하기를 원할 것임.

- <strong>개인 맞춤형 스킬:</strong>

- 회사 내에서도 직원들은 자신의 특정 작업 방식과 워크플로우에 맞게 스킬을 맞춤 설정하는 것이 합리적임.

- 예: 한 영업 담당자의 카피라이팅 스타일은 다른 담당자와 다를 수 있으므로, 자신만의 카피라이팅 스타일에 맞게 스킬을 맞춤 설정할 수 있음.

- 개인이 스킬 구축에 능숙해지고 특정 도메인 전문 지식을 바탕으로 스킬을 구축한다면, 일반 대중에게 제공하여 수익화할 가능성도 있음.

## **스킬 구축 방법**

- 스킬을 구축하는 두 가지 주요 방법이 있음:
        <ol>

- AI 에이전트와 함께 수동으로 작업을 한 번 수행한 다음, 이를 스킬로 저장하도록 요청하는 방법.

- 즉시 스킬을 구축하는 방법을 에이전트에게 지시하는 방법.
        </ol>

- 예시: 타사 메타 광고 스킬을 다운로드하여 가져온 후, 이 스킬과 특정 지식 소스를 결합하여 새 광고 생성 스킬을 만들도록 요청한 다음, 이를 라이브러리에 추가하는 과정이 있음.

- Entropic 내장 플러그인 및 스킬 맞춤 설정은 매우 쉬움. 'Skills'로 이동하여 'Edit'을 클릭하고 Claude로 맞춤 설정하거나, 플러그인에서 'Customize'를 클릭하면 됨.

- 프롬프트를 통해 참조 파일을 제공하여 자신만의 스킬을 생성할 수 있음.

## **스킬 구축을 위한 모범 사례 (스킬 엔지니어링)**

- 좋은 스킬을 구축하는 것은 예술과 같음. 프로세스에 대한 도메인 전문 지식을 에이전트가 사용할 수 있는 제품으로 만드는 예술임.

- 스킬 엔지니어링은 소프트웨어 엔지니어링과 많은 유사점을 가지고 있지만, AI 에이전트를 위한 소프트웨어 엔지니어링과 같음.

- 사용자 경험(UX), 즉 사람의 개입을 언제 추가할지 고려해야 함.

- 컨텍스트 엔지니어링, 즉 어떤 컨텍스트 균형이 최상의 결과를 생성하는지 고려해야 함.

- 기능 추가, 제거, 엣지 케이스 처리 등을 생각해야 함.

- 스킬의 장점은 프롬프트로 쉽게 업데이트할 수 있어 스킬이 결코 완성되지 않는다는 점임. 많이 사용할수록 더 나아짐.

- 인포그래픽 스킬의 경우, 원하는 결과를 얻기 위해 약 5번 업데이트하고 반복함.

- <strong>첫 번째 단계: 계획 (대부분의 사람들이 건너뛰지만 가장 큰 영향을 미침)</strong>

- 스킬을 구축하거나 맞춤 설정하기 위해 프롬프팅하기 전에, 잠시 시간을 들여 좋은 결과물을 얻기 위한 이상적인 단계별 프로세스를 숙고해야 함.

- AI 에이전트에게 더 나은 작업을 수행하도록 어떤 지식 소스나 추가 정보를 제공할 수 있는지 생각해야 함.

- <strong>추천하는 일반적인 컨텍스트 또는 참조 파일 (특히 마케팅 또는 영업 관련 스킬의 경우):</strong>

- 우리가 하는 일: 비즈니스 설명, ICP 설명, 목소리 개성 (LinkedIn 등), 뉴스레터 전략, 유튜브 전략, 글쓰기 프레임워크 등.

- 이러한 문서는 여러 스킬에 걸쳐 재사용할 수 있음.

- 이러한 참조 파일이 없다면, Claude와 함께 만들 수 있음.

- 예: 유튜브 전략 문서를 만들도록 요청하고, Claude가 유튜브 전략에 대한 질문을 한 후, 계획 모드를 통해 더 많은 컨텍스트를 얻어 유튜브 전략 문서를 생성할 수 있으며, 이를 여러 유튜브 스킬에 재사용할 수 있음.

- 이러한 파일이 몇 개 있으면 스킬 구축 효율성이 훨씬 높아짐.

- 이후 에이전트가 이 작업을 수행하는 데 필요한 도구나 소프트웨어가 무엇인지 고려해야 함.

- 가능하다면 항상 좋은 출력 예시를 준비해야 함. 이것이 일반적으로 성능에 가장 큰 영향을 미침.

- <strong>두 번째 단계: 구축 (프롬프팅)</strong>

- 다음은 좋은 첫 스킬 결과물을 얻기 위해 포함할 내용을 고민하는 데 도움이 될 수 있는 프레임워크임.

- <strong>스킬 이름 및 트리거 방법 정의:</strong>

- 예: 스킬 이름은 '인포그래픽 생성기'이며, 사용자가 인포그래픽 생성을 언급할 때 트리거되어야 함. 에이전트는 이를 사용하여 메타 설명을 작성함.

- <strong>스킬의 목표 또는 목적 정의:</strong>

- 이 부분은 나중에 프로세스에서 더 자세히 다루므로 짧아도 됨.

- 예: LinkedIn 및 뉴스레터용으로 내 브랜드 스타일에 맞는 고품질 인포그래픽을 생성하는 스킬.

- <strong>Claude에게 필요한 커넥터, API 또는 MCP 컨텍스트 제공:</strong>

- 이 스킬에 대해 특정 테이블, 페이지 또는 소프트웨어 내에서 따라야 할 프로세스가 있다면 여기에 설명할 수 있음.

- <strong>단계별 프로세스 명시 (가장 중요하며 시간을 들여야 할 부분):</strong>

- 각 단계에 무엇을 포함할지 고려하는 좋은 방법은 다음과 같음:

- 무엇을 해야 하는가?

- 언제 사람의 개입이 필요한가? 어떤 종류의 사람 개입인가? (Claude의 동적 QA 박스를 통해 체크박스, 오픈 필드, 단일 선택 등을 선택할 수 있으며, 이는 UX 디자인과 유사함.)

- 이 작업을 더 잘 수행하기 위해 어떤 추가 컨텍스트를 사용해야 하는가?

- 내용이 매우 짧다면 프롬프트에 추가할 수 있음. 하지만 에이전트가 읽어야 할 특정 파일이 있다면, 각 단계에 대해 명시해야 함.

- 참조 파일이 아직 없더라도 컨텍스트를 프롬프트에 붙여넣고 해당 스킬에 대한 참조 파일을 만들도록 요청할 수 있음.

- 일반적으로 skill.md는 프로세스에 집중하여 매우 깔끔하게 유지해야 함. 추가 정보는 참조 파일에 있어야 스킬 성능이 훨씬 향상됨.

- <strong>각 단계에 대한 예상 출력 정의:</strong>

- 사람 개입 단계에서는 Claude에게 항상 여러 변형 또는 옵션을 제공하도록 요청하는 것이 생산성에 큰 영향을 미침. 단일 출력 대신 선택할 수 있는 여러 옵션을 제공하도록 지시함.

- 예: 인포그래픽이 아이디어나 개념을 시각화하는 다섯 가지 다른 방법을 제공해야 함.

- <strong>스킬 규칙 설정:</strong>

- 스킬 실행 시 발생할 수 있는 잠재적인 문제점을 예측하고 규칙으로 작성해야 함.

- 이 섹션은 스킬 개선 시 지속적으로 업데이트될 부분임.

- 두 가지 팁:
                        <ol>

- 필요할 때 지식 파일을 의무적인 단계로 사용하도록 지시하는 것을 강조해야 함 (그렇지 않으면 건너뛰는 경향이 있음).

- 사람 개입 시 여러 변형 제공을 다시 한번 강조함.
                        </ol>

- <strong>점진적 업데이트 포함 (자기 학습):</strong>

- 스킬을 사용할 때 자동으로 업데이트되고 개선되도록 지시하여 자기 학습이 가능하도록 해야 함.

- 예: 이 스킬에서 더 이상 하지 말아야 할 명확한 사항을 정의할 때마다 규칙 섹션을 업데이트하도록 지시함.

- 사용자가 최종 결과물을 승인하면, 이를 좋은 예시로 저장하여 좋은 것이 무엇인지에 대한 데이터를 자동으로 학습하고 구축하도록 프로세스에 지시하는 것이 매우 강력할 수 있음.

## **인포그래픽 스킬 데모 및 개선 지침**

- 인포그래픽 스킬이 어떻게 작동하는지 시연함.

- 코드 탭을 사용하여 슬래시 명령으로 스킬을 트리거함. 이는 초기 프롬프트에서 업데이트된 버전임.

- <strong>데모 과정:</strong>
        <ol>

- 어떤 콘텐츠를 인포그래픽으로 전환할 것인지 질문하며, 스킬에 대한 설명 부분을 붙여넣음.

- 나중에 추가된 업데이트로, 어떤 플랫폼용인지 먼저 질문하여 어떤 형식으로 생성해야 할지 파악함 (예: LinkedIn). QA 박스를 사용하여 지시한 대로 답변을 받음.

- 어떤 유형의 시각 자료인지 질문함 (예: 인포그래픽).

- 나중에 체크박스를 통해 여러 변형을 생성할 수 있도록 조정 및 반복함. 시각화할 내용을 제안하고 (예: 스킬의 해부학), 해당 개념을 시각화하는 방법을 제안함.

- 원하는 것을 선택하거나 모두 생성할 수 있음.
        </ol>

- 첫 프롬프트는 이 정도로 좋지 않았고, 세네 번의 반복, 추가 규칙 등을 통해 개선되었으며, 이터레이션 프로세스였음. 많이 사용할수록 더 좋아진다는 점이 중요함.

- 결과물은 화자의 브랜드 스타일과 가이드라인을 잘 따름.

- 마지막에 '모두 보관 및 마무리'를 클릭하면 좋은 출력 예시로 저장되어 모델이 좋은 것이 무엇인지 학습하도록 훈련됨.

- <strong>모델이 변경 사항을 적용하도록 지시하는 방법:</strong>

- <strong>일반적인 개선 및 업데이트 지침:</strong>

- <strong>프로세스를 올바르게 따르지 않는 경우:</strong> skill.md를 변경하도록 요청함. skill.md는 많은 추가 정보로 오염되지 않고 핵심 프로세스에 집중해야 함.

- <strong>추가 정보가 필요한 경우:</strong> 참조 파일을 추가하도록 요청함.

- <strong>특정 작업을 절대 하지 않기를 원하는 경우 (예: 검은색 배경 사용 금지):</strong> 규칙을 추가하거나 지식 파일을 업데이트하도록 요청함.

- <strong>소프트웨어, MCP, 도구 사용에 어려움을 겪는 경우:</strong> 먼저 수동으로 필요한 특정 작업을 안내한 다음, 소프트웨어에서 해당 작업을 수행하는 방법에 대한 MCP 참조 문서를 만들도록 요청함.

- 첫 버전과 최종 버전은 완전히 다름을 강조함.

## **스킬 및 플러그인 공유 방법**

- 스킬이 잘 작동하고 마켓플레이스나 팀원과 공유하고 싶을 때 가장 쉬운 방법은 Claude에게 해당 스킬의 ZIP 파일을 요청하는 것임.

- 이 ZIP 파일을 공유하면 다른 사람이 설정, 기능에서 ZIP 파일을 업로드할 수 있음.

- 스킬을 GitHub를 통해 배포할 수도 있음 (링크는 설명에 포함).

- 여러 스킬을 가지고 있고 다른 스킬, 명령, 에이전트, 특정 커넥터를 결합하고 싶다면, 이를 플러그인으로 묶을 수 있음.

- 플러그인 구축은 Claude에게 플러그인을 만들도록 요청하면 되며, 포함할 모든 스킬 등을 질문한 다음 라이브러리에 추가할 수 있음.

- 플러그인도 ZIP 파일이나 GitHub 업로드를 통해 공유 가능함.

- 비즈니스에서 여러 부서에 걸쳐 여러 플러그인이 있다면, 모든 플러그인을 묶은 플러그인 마켓플레이스를 만들 수 있음.

- 이것은 Claude 코드와 함께 생성하고 GitHub에 업로드하여 링크를 얻어야 함 (이에 대한 전체 가이드도 링크로 공유됨).
