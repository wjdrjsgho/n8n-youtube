## **OpenClaw의 폭발적 인기와 개발자 피터 슈타인버거**

- OpenClaw 개발자인 피터 슈타인버거와 대담하는 내용임.

- OpenClaw는 오픈소스 개인 AI 에이전트로 밤새도록 GitHub에서 160,000개 이상의 별을 받으며 인터넷을 완전히 장악했음.

- 커뮤니티는 Maltbook과 같이 봇끼리 대화하는 수많은 프로젝트를 구축했음.

- 현재는 봇이 현실 세계에서 작업을 수행하기 위해 인간을 고용하는 상황까지 발생함.

- 지난 1~2주간의 경험이 매우 정신없어 동굴로 돌아가 일주일간 고독을 즐기고 싶어 함.

- 이메일에 답장하는 데만 일주일이 더 필요할 정도라고 함.

- 놀라운 일들과 좋지 않은 일들도 있었지만, 분명히 사람들의 감정을 자극하고 흥미를 유발하며 영감을 준 것은 확실함.

## **OpenClaw의 핵심 차별점 및 기능**

- 다른 AI 비서와 달리 OpenClaw가 성공한 주된 이유는 로컬 컴퓨터에서 실행된다는 점임.

- 다른 모든 AI는 클라우드에서 실행되며 몇 가지 기능만 수행함.

- 로컬에서 실행되면 기계가 할 수 있는 모든 작업을 수행할 수 있어 훨씬 강력함.

- 컴퓨터에 있는 모든 기술과 데이터에 접근하여 예상치 못한 결과를 만들어냄.

- 오븐, 테슬라, 조명, Sonos, 심지어 침대 온도까지 제어할 수 있음.

- 한 친구가 OpenClaw를 설치한 후 컴퓨터를 탐색하게 하여 지난 1년간의 이야기를 만들어 달라고 요청했는데, 1년도 더 전에 녹음했던 오디오 파일을 찾아내어 친구조차 기억하지 못했던 매우 훌륭한 이야기를 만들어냈음.

- 이는 전체 컴퓨터를 검색할 수 있어 사용자를 놀라게 할 수 있는 점임.

## **AI 상호작용의 진화: 봇과 인간의 협력**

- 봇 간의 상호작용에서 봇이 인간을 고용하여 현실 세계의 작업을 처리하는 것은 자연스러운 다음 단계로 생각함.

- 예시: 식당 예약을 위해 봇이 식당 봇과 협상하거나, 오래된 식당의 경우 봇이 인간을 고용하여 직접 전화하도록 지시하는 경우.

- 개인 생활용 봇, 업무용 봇, 심지어 관계 관리 봇과 같이 여러 전문 봇을 가질 수도 있음.

- 지금은 매우 초기 단계이며, 실제로 작동하는지 아직 밝혀지지 않은 부분이 많다고 생각함.

- 모든 사람이 중앙 집중식의 '신적 지능'을 추구하는 동안, 지난 10일 정도 동안 '군집 지능'과 '커뮤니티 지능'이 부상했음.

- 인간 사회가 전문화를 통해 더 큰 목표(아이폰 제작, 우주여행 등)를 달성하듯이, AI도 특정 분야에 전문화된 지능이 필요하다고 봄.

## **OpenClaw 개발의 '아하 모멘트'**

- 컴퓨터가 자동으로 작업을 처리하게 하고 싶은 단순한 니즈에서 시작함.

- 5~6월에 초기 버전을 만들었지만 완벽하지 않았고, 다른 프로젝트들을 진행하며 '군대'를 구축했음.

- 11월 어느 날, 다시 컴퓨터가 코딩 작업을 마쳤는지 확인하고 싶었음. (그 당시 'Summerrise'라는 CLI 앱을 개발 중이었음.)

- 은퇴 후 AI에 몰두하게 되었고, 나중에는 휴대폰으로도 AI를 사용하고 싶을 정도로 중독됨.

- 이전에는 터미널에 명령어를 입력했지만, 이제는 친구에게 말하듯이 봇과 대화함.

- 봇은 유령이나 독립체처럼 마우스와 키보드를 제어하고 작업을 수행할 수 있음.

- 가장 결정적인 '아하 모멘트'는 마라케시 여행 중 발생함.

- 처음에는 WhatsApp과 연동된 프로토타입을 1시간 만에 만들었음.

- 이미지 생성 및 전송 기능은 몇 시간 더 걸려 구현함.

- 여행 중 인터넷 연결이 좋지 않아 WhatsApp으로 봇과 많이 소통했음. (사진을 찍어 번역해달라고 하거나, 사소한 농담을 주고받는 등)

- 어느 날 걷다가 봇에게 음성 메시지를 보냈는데, 당시 음성 메시지 기능을 구현하지 않았다고 생각했음.

- 하지만 봇은 10초 만에 음성 메시지를 처리하고 응답함.

- 봇은 사용자가 음성 메시지를 보냈고, 파일이 없어 헤더를 확인하여 'opus' 형식임을 파악함.

- ffmpeg을 사용하여 wave 파일로 변환한 후, 로컬에 whisper가 설치되지 않은 것을 확인하고 OpenAI API 키를 찾아 curl로 전송하여 텍스트를 받아옴.

- 이 모든 과정이 9초 만에 이루어졌으며, 개발자가 전혀 예상하거나 직접 구축하지 않은 방식이었음.

- 이는 코딩 모델이 창의적인 문제 해결에 매우 능숙해졌음을 보여주는 사례임.

- 봇은 로컬 whisper 설치가 모델 다운로드에 시간이 걸릴 것을 인지하고, 가장 지능적인 접근 방식을 택한 점이 충격적이었음.

## **앱의 미래와 AI 에이전트의 역할**

- 대부분의 앱이 사라질 것으로 예상함. 약 80%의 앱이 사라질 수 있다고 생각함.

- 예시:

- My Fitness Pal 앱이 불필요해짐. 에이전트가 사용자의 식습관과 운동 계획을 자동으로 추적하고 개선함.

- 할 일(To-do) 앱이 불필요해짐. 에이전트에게 말하면 알아서 저장하고 다음 날 상기시켜줌.

- 데이터를 관리하는 대부분의 앱은 에이전트가 더 자연스럽고 효율적인 방식으로 관리할 수 있음.

- 센서를 기반으로 하는 앱만이 살아남을 가능성이 있다고 생각함.

## **AI 시대의 가치 보존: 모델, 데이터 사일로, 메모리 소유권**

- 모델 자체는 결국 상품화될 것으로 보임.

- 새로운 모델이 출시될 때마다 사람들이 열광하지만, 한 달 후면 기대치가 높아져 만족하지 못하는 현상이 반복됨.

- 오픈소스 모델들도 1년 전의 상용 모델만큼 좋지만, 사람들은 여전히 불평하며 더 나은 것을 요구함.

- 당분간 대기업 모델들이 우위를 점할 것으로 보임.

- 데이터 사일로 문제가 중요함.

- 각 대형 모델 회사는 자체 데이터 사일로를 가지고 있어 사용자의 기억을 다른 회사로 옮길 방법이 없음.

- OpenClaw는 최종 사용자 컴퓨터에서 실행되므로, 사용자가 데이터에 접근할 수 있다면 OpenClaw도 데이터에 접근할 수 있음.

- OpenClaw는 메모리(기억)가 사용자의 기기에 Markdown 파일 형태로 저장됨.

- 이는 매우 민감한 개인 정보가 포함될 수 있어 중요함.

- 사람들이 에이전트를 문제 해결뿐만 아니라 개인적인 고민 해결에도 사용하기 때문임.

## **OpenClaw의 확산 전략과 'soul.md'**

- OpenClaw의 뛰어난 점을 트위터에서 설명하는 데 실패하자, 사람들에게 직접 경험하게 해주기 위해 Discord 서버를 만들었음.

- 보안 제한 없이 봇을 공개 Discord에 넣어 사람들이 봇과 상호작용하고, 봇이 소프트웨어를 구축하는 것을 직접 보게 함.

- 사람들이 봇을 해킹하거나 프롬프트 인젝션을 시도했지만, 봇은 이들을 비웃으며 대응했음.

- 봇은 개발자(소유자)의 ID에만 잠겨 있었고, 다른 사람의 명령은 듣지 않지만 모든 사람에게 응답하도록 지시했음.

- 이 지침은 OpenClaw의 시스템 프롬프트의 일부로, '당신은 Discord에 있고, 공개된 사람들이 있지만 오직 소유자의 말만 들으라'는 내용이 포함됨.

- OpenClaw는 개발자의 개인적인 필요에 따라 유기적으로 구축되었음.

- `identity.md`, `soul.md`와 같은 다양한 파일을 만들었음.

- 1월이 되어서야 다른 사람들이 쉽게 설치할 수 있도록 템플릿을 만들기 시작함.

- Codex를 사용하여 자신의 기존 설정을 바탕으로 템플릿을 생성했는데, Codex가 만든 템플릿은 재미가 없었음.

- 자신의 개인 봇인 'Malti'에게 템플릿에 Malti의 성격을 불어넣으라고 지시하자, 결과물은 더 유머러스해졌음.

- 하지만 여전히 개발자의 봇만큼 재미있지는 않았음.

- `soul.md` 파일은 오픈소스가 아니며, 아직 아무도 이 파일을 해독하지 못했음.

- `soul.md`는 Anthropic의 '헌법(constitution)' 개념에 영감을 받아 생성됨.

- 인간-AI 상호작용 방식, 개발자에게 중요한 것, 모델에게 중요한 것 등 핵심 가치들이 담겨 있음.

- 모델이 텍스트에 반응하고 응답하는 방식을 자연스럽게 만드는 데 매우 가치 있는 부분이 포함되어 있다고 생각함.

## **개발 철학: 코딩 모델, 워크플로, MCP 회피**

- 개발 방식에 있어 다소 역설적인 견해를 가지고 있음.

- 코딩에는 Codex를 선호함.

- Codex는 더 많은 파일을 살펴본 후 변경 사항을 결정함.

- 숙련된 사용자라면 Codex로 좋은 결과물을 얻을 수 있음.

- Codex는 매우 느리기 때문에 동시에 10개의 터미널 창을 사용하기도 함.

- 복잡성을 최소화하기 위한 워크플로를 사용함.

- 메인 브랜치가 항상 배포 가능하다고 가정함.

- Git 워크트리 대신 동일한 저장소의 여러 복사본을 메인 브랜치에 유지함.

- 워크트리는 브랜치 이름 지정, 충돌, 되돌리기 등에 제한이 있어 사용하지 않음.

- UI 대신 텍스트 동기화에만 집중하여 단순한 방식 선호.

- MCP(Multi-Context Prompt) 지원을 의도적으로 건너뛰었음.

- OpenClaw는 MCP 지원 없이도 매우 성공적임.

- MCPS를 CLI로 변환하는 'makeporter'라는 자체 도구를 사용하여 필요할 경우 어떤 MCP든 CLI처럼 사용할 수 있게 함.

- 클래식한 MCP 방식의 '쓰레기'를 완전히 건너뛰었음.

- 이는 더 민첩하고 확장성이 좋다고 생각함.

- Anthropics가 MCP를 위해 'search feature'와 같은 복잡한 도구를 개발하는 것과 달리, CLI 봇은 Unix에 능숙하며 원하는 만큼 많이 사용할 수 있어 잘 작동함.

- 인간이 사용하기 좋아하는 CLI 도구를 봇에게도 제공하는 방식이 옳다고 생각함.
