## **<strong>개발자의 역할 변화: 코드 작성자에서 마크다운 문서 작성자로</strong>**

- 개발 현장에서 코드를 직접 타이핑하는 시간보다 마크다운 문서를 정리하는 시간이 더 길어졌음.

- 이러한 변화가 왜 발생했고, 무엇이 바뀌었으며, 이 변화에 어떻게 대응해야 하는지에 대해 설명함.

## **<strong>개발의 발전 방향: '덜 짜는' 추상화</strong>**

- 개발의 역사는 항상 '덜 짜는' 방향으로 발전해 왔음.

- <strong>과거의 추상화 단계별 발전 과정 및 저항:</strong>

- <strong>1950년대 기계어 (어셈블리어):</strong>

- CPU에 0과 1을 직접 나열하여 명령을 내리던 시절임.

- 화면에 '헬로우' 하나 띄우려면 이진수를 줄줄이 써야 했음.

- 당시 어셈블리 개발자들은 C 언어가 나오자 "이건 진짜 프로그래밍이 아니다", "기계를 직접 다루지 않으면 개발자가 아니다"라고 반발했음.

- 결국 C 언어가 표준이 되었음.

- <strong>C 언어의 등장:</strong>

- `printf("Hello")` 한 줄로 끝나는 시대로, 하드웨어 구조를 몰라도 프로그래밍이 가능해졌음.

- <strong>C++ 및 Java의 등장 (객체 지향):</strong>

- 객체 지향 개념 도입으로 현실 세계 모델링이 가능해졌음.

- 메모리 직접 관리(malloc, free) 대신 Java의 가비지 컬렉터가 자동 처리함.

- 당시 C 개발자들은 "메모리 직접 관리 안 하는데 그게 무슨 개발이냐"며 반발했음.

- 결국 C++과 Java가 C의 영향력을 대체했음.

- <strong>Python의 등장:</strong>

- `print("Hello")` 한 줄이면 끝나고, 타입 선언이나 세미콜론도 필요 없는 거의 영어 문장 같은 언어임.

- 당시 개발자들은 "파이썬은 장난감 언어일 뿐, 실제 개발에는 쓸 수 없다"며 반발했음.

- 현재 파이썬은 전 세계에서 가장 많이 사용되는 언어가 되었음.

- <strong>반복되는 패턴:</strong>

- 매번 추상화 수준이 올라갈 때마다 기존 개발자들은 "이건 진짜 개발이 아니다"라고 반발했음.

- 그러나 매번 더 높은 추상화 수준 (즉, 더 쓰기 쉬운 방식)이 표준이 되었음.

- <strong>현재의 흐름:</strong>

- 이제 "Hello를 출력해 줘"와 같은 자연어 한 마디가 코드가 되는 시대가 도래했음.

- 이는 기계어에서 C, C++, Java, Python으로 이어진 개발 발전 방향의 연장선으로 볼 수 있으며, 갑작스러운 변화가 아님.

## **<strong>AI 코딩 도구의 실제 작동 방식 및 마크다운의 역할</strong>**

- <strong>Kloud Code 활용 예시:</strong>

- Kloud Code를 사용할 때 가장 먼저 프로젝트 폴더에 `kloud.md`라는 마크다운 파일을 만듦.

- 이 파일에 프로젝트의 전체 맥락(예: Angular 18 사용, Firebase 백엔드, Z-build 빌드, 배포 방식 등)과 평소 워크플로우를 상세히 기록함.

- AI는 이 파일을 읽어 프로젝트의 맥락을 이해한 후 코드를 작성함.

- 예시: 새로운 API 엔드포인트 추가 요청 시, AI는 `kloud.md`를 보고 파이어베이스 클라우드 펑션, 파이썬 기반, Ruff 린팅 등 프로젝트 컨벤션을 파악하여 이에 맞춰 코드를 생성함. 개발자가 일일이 설명할 필요가 없음.

- <strong>다른 AI 도구에서의 유사성:</strong>

- Cursor 도구에는 `cursor_rules` 파일이 존재함.

- GitHub Copilot에는 `copilot_instruction.md` 파일이 존재함.

- Windf에는 `windf_rules` 파일이 존재함.

- 이름은 다르지만, 모두 마크다운 파일이며 프로젝트 규칙이나 스타일 가이드를 담아 AI가 참고하도록 하는 동일한 역할을 함.

- <strong>Kloud Code의 '스킬(Skill)' 개념:</strong>

- 겉보기에는 대단한 기능처럼 보이나, 실제로는 마크다운 파일들을 모아 놓은 서브 폴더에 불과함.

- AI가 특정 상황에서 참고해야 할 지시 사항을 마크다운으로 작성해 놓은 파일 묶음임.

- <strong>MCP (Model Context Protocol)와의 비교:</strong>

- MCP는 AI가 외부 시스템(데이터베이스, 외부 API 등)과 통신하기 위한 표준 규격임 (예: USB-C와 유사).

- 스킬(마크다운 파일)은 MCP와 달리 철저히 내부적인 것이며, AI가 특정 작업을 할 때 어떤 맥락, 규칙, 패턴을 따라야 하는지를 알려주는 컨텍스트 문서의 집합임.

- 결론: AI 코딩 도구에서 가장 강력한 기능 중 하나는 복잡한 시스템이나 특별한 언어가 아닌, 바로 마크다운 폴더와 자연어임. 이것이 현재 패러다임 전환의 핵심임.

## **<strong>워크플로우 설계의 변화: 자연어 지시의 시대</strong>**

- 단순히 코드를 생성하는 것을 넘어, 작업의 흐름(워크플로우)을 설계하는 방식 자체가 바뀌었음.

- <strong>기존 워크플로우 자동화 방식:</strong>

- Airflow나 LangChain 같은 프레임워크를 사용하여 코드로 파이프라인을 구축했음.

- 함수 실행 순서, 조건부 로직, 에러 처리 등을 모두 코드로 구현했음.

- <strong>AI 기반의 새로운 워크플로우 자동화 방식:</strong>

- 마크다운 파일에 "이거 해. 그다음에 이거 하고, 마지막에 이거 확인해서 이거면 저거 하고 아니면 이렇게 끝내"와 같이 자연어로 순서만 적어 놓음.

- AI가 이를 읽고 순서대로 실행하며, 돌발 상황이나 변수 발생 시 유저에게 물어보는 등 유연하게 대처함.

- 최종 결과물을 도출함.

- <strong>변화의 중요성:</strong>

- 과거에는 파이프라인을 짜려면 파이썬 코드 (Airflow의 DAG, LangChain의 체인)가 필요했음.

- 이제는 마크다운에 한국어로 순서를 적는 것 자체가 코드를 대체할 수 있게 되었음.

- 코드로 플로우를 설계하고 구현하던 시대에서 자연어로 플로우를 지시하는 시대로 완전히 넘어왔음.

## **<strong>개발자의 본질적 역할 변화: 컨텍스트 설계자</strong>**

- 개발자의 본질은 코드를 직접 타이핑하는 사람이 아니라, AI가 올바른 코드를 만들 수 있도록 컨텍스트를 잘 설계하는 사람으로 바뀌었음.

- 핵심 전환: 코드를 잘 짜는 능력에서 AI가 잘 이해할 수 있도록 문서를 잘 쓰는 능력, 더 깊이 들어가면 컨텍스트를 잘 설계하는 능력으로 개발자의 역할이 변화함.

- <strong>컨텍스트 설계의 중요성 비유:</strong>

- <strong>건축가 비유:</strong>

- 건축가는 직접 벽돌을 쌓지 않고 설계도를 그림.

- 설계도가 정확해야 건물이 제대로 올라가며, 엉망인 설계도는 아무리 훌륭한 인부와 장비가 있어도 건물을 무너뜨림.

- 개발자는 이제 코드라는 벽돌을 직접 쌓기보다, 컨텍스트라는 설계도를 작성하는 사람이 되었음.

- <strong>영화 감독 비유:</strong>

- 영화 감독은 직접 연기하지 않고 배우에게 의도와 디렉션(예: "이 장면에서 분노하고 있지만 속으로는 웃고 있어야 해")을 전달함.

- `kloud.md`와 같은 마크다운 파일이 바로 이 대본 역할을 함.

- 이 대본의 품질이 AI가 만드는 코드의 품질을 결정함.

- <strong>실제 개발 현장 변화 (Meta 경험):</strong>

- 이전에는 PR(Pull Request) 리뷰 시 '이 코드를 누가 짰는가'가 중요했지만, 이제는 AI가 짠 코드가 사람의 터치 없이 그대로 PR에 올라오는 것이 일상이 되었음. 회사에서도 이를 장려함.

- 중요해진 것은 '이 코드의 의도가 무엇인가', '어떤 컨텍스트에서 이런 결정이 내려졌는가'임.

- 코드 자체보다 그 코드 뒤에 있는 설계와 의도(인텐트)가 더 중요해졌음.

## **<strong>기초 지식의 중요성: AI 시대의 역설적 경쟁력</strong>**

- <strong>흔한 오해와 그 위험성:</strong>

- "이제 코드를 안 배워도 되겠네. AI 시키면 되니까"라는 착각을 할 수 있음.

- "비전공자가 AI로 앱 만들고 코드 한 줄 안 쓰고 서비스 런칭" 같은 글들은 데모까지는 쉽게 만들 수 있음을 보여주지만, 실제 서비스는 전혀 다른 문제임.

- 보안, 동시 접속 처리, 데이터베이스 스키마 설계, 캐싱 등은 코드를 이해하지 못하면 문제 자체를 인지하거나 질문할 수 없음.

- <strong>기초 지식의 지속적인 중요성:</strong>

- <strong>건축가 비유:</strong> 건축가가 구조 역학을 모르면 설계 자체를 할 수 없으며, 아무리 멋진 도면도 하중 계산이 잘못되면 건물이 무너짐.

- <strong>개발자 비유:</strong> 개발자도 코드를 이해하지 못하면 좋은 컨텍스트를 만들 수 없음.

- AI가 짠 코드가 맞는지 검증하고, 잘못된 방향으로 갈 때 바로잡으려면 여전히 기본기가 중요함.

- <strong>결론:</strong>

- 변하고 있는 것은 개발자가 코드를 직접 타이핑하는 비중이며, 이는 확실히 줄어들고 있음 (발화자는 3개월 전부터 코드를 직접 타이핑한 적이 거의 없음).

- 하지만 변하지 않는 것은 시스템을 이해하는 능력임.

- 역설적으로 AI가 더 많은 것을 해줄수록, 이런 기초 지식을 가진 사람들의 경쟁력이 더욱 올라감.

- AI를 도구로 쓸 수 있는 사람과 없는 사람의 격차가 점점 벌어지며, 도구를 제대로 쓰려면 결국 기본기가 필요함.

## **<strong>결론 및 개발자 스킬셋의 변화</strong>**

- 개발의 역사를 보면 추상화는 항상 더 높은 방향으로, 더 쉬운 방향으로 진행되어 왔음 (기계어 -> C -> C++ -> Python -> 자연어 및 컨텍스트 문서 코딩).

- 매번 "그건 진짜 개발이 아니다"라는 반발이 있었지만, 매번 그 주장은 틀렸음.

- 개발자라는 직업이 사라지는 것은 아니지만, 개발자에게 요구되는 스킬셋의 무게 중심이 확실히 바뀌고 있음을 인정해야 함.

- 코드를 직접 짜는 능력보다 AI가 이해할 수 있도록 컨텍스트를 잘 설계하는 능력이 앞으로 필수적인 능력이 될 것임.

- 각자의 `kloud.md` 또는 `cursor_rules` 파일이 잘 정리되어 있는지 다시 한번 확인할 필요가 있음.
