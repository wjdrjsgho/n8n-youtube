## **코덱스(Codex) 소개 및 활용의 필요성**

- 기존 채팅 기반 AI 툴(ChatGPT, Gemini) 사용 중인 경우가 많음.

- Claude Code 같은 툴에 대한 관심이 있으나 추가 구독 비용에 부담을 느끼는 경우가 있을 수 있음.

- 오픈AI의 코덱스(Codex)를 사용해 볼 필요가 있음.

- 코덱스는 오픈AI 버전의 Claude Code라고 생각하면 됨.

- 코딩 작업 외에 일반 사무 업무에서도 ChatGPT보다 강력한 성능을 발휘함.

- <strong>기존 AI 툴의 문서 작업 한계점</strong>

- ChatGPT나 Gemini로 문서 작업을 할 수는 있음.

- 프롬프트로 작업을 요청하고 아이디어를 얻어 결과물을 복사 붙여넣기 하거나, 커넥터 등을 활용해 문서 결과물을 바로 생성하는 작업도 가능함.

- 그러나 대부분의 사무 업무는 단순히 하나의 결과물 생성에 그치지 않음.

- 여러 문서를 참조하여 특정 결과물을 생성하고, 해당 결과물을 반복적으로 생성 및 관리해야 할 필요가 있음.

- <strong>코덱스의 프로젝트 관리 적합성</strong>

- 프로젝트 관리 측면에서 ChatGPT보다 코덱스가 더 적합한 툴임.

- 작업 폴더를 생성해 두고 PPT 파일, 이미지 파일, 검수 리포트 등을 한눈에 관리할 수 있음.

- <strong>코덱스 사용 비용 효율성</strong>

- ChatGPT 구독을 통해 코덱스를 활용할 수 있어, 추가 구독 비용 부담이 없음.

- <strong>실무 활용 예시</strong>

- 제안서 PPT 제작을 예시로 코덱스 앱의 실무 활용 방법을 알아볼 예정임.

- 에이전팅 AI 툴을 활용하여 업무 효율을 높이고 싶은 사람들에게 도움이 될 것임.

## **코덱스(Codex) 설치 및 초기 설정**

- <strong>코덱스 설치 방법</strong>

- ChatGPT에 접속하면 왼쪽에 코덱스 섹션이 있음.

- 코덱스 섹션을 누르면 세 가지 방식으로 코덱스를 설치할 수 있음.

- 이 중 앱 다운로드 버전을 추천함.

- Claude Code의 경우 CLI를 주로 활용하지만, 코덱스는 앱이 더 잘 만들어져 있어 사용성이 좋았음.

- 처음 시작하는 경우 앱으로 시작하는 것을 추천함.

- 코덱스 앱은 Mac과 Windows를 모두 지원함.

- '컴퓨터 유즈(Computer Use)' 같은 일부 기능은 아직 Mac 버전만 제공하는 것 같으므로, 이 부분은 염두에 둘 필요가 있음.

- 표시된 OS별 다운로드 버튼을 눌러 다운로드 후 코덱스 앱으로 진입함.

- <strong>로그인 및 초기 화면 구성</strong>

- 코덱스 앱 실행 후 ChatGPT 계정으로 로그인함.

- 초기 화면은 ChatGPT와 유사하게 나타남.

- 가장 큰 차이점은 코덱스가 프로젝트 폴더 기반으로 작동한다는 점임.

- 채팅 화면도 있으나, 기본적으로 프로젝트 폴더를 선택하도록 되어 있음.

- 해당 폴더는 특정 작업을 할 때 관련된 문서들이 저장되는 공간으로 이해하면 됨.

- 따라서 코덱스 활용 시 작업 공간을 먼저 세팅하는 것이 중요함.

- <strong>프로젝트 폴더 구조 설정 예시</strong>

- '코덱스 PPT 데모(Codex PPT Demo)'라는 상위 디렉토리를 생성할 예정임.

- 그 안에 '에이전트(Agent)', '디자인(Design)', '비즈니스 컨텍스트(Business Context)', '아웃풋(Output)' 등의 폴더를 생성함.

- 작업 요청 전 기본적으로 세팅해야 하는 문서들(agent.md, design.md, business_context.md, deck_brief.md)을 생성함.

- <strong>상위 디렉토리 생성</strong>

- 왼쪽 프로젝트 섹션에서 '새로운 프로젝트 추가'를 선택함.

- 'Existing Folder'를 눌러 로컬 디렉토리에서 새 폴더를 생성하고 지정하여 활용할 수 있음.

- '코덱스 PPT 데모' 폴더를 생성하고 선택함.

- <strong>Git을 활용한 버전 관리 시스템 설정</strong>

- 코덱스에서 작업을 시작하기 전에 작업 폴더를 Git으로 관리하는 것이 좋음.

- <strong>Git 설명:</strong> 폴더 내 작업 시 파일 변경 사항을 저장하고, 필요시 이전 버전으로 원복할 수 있도록 버전 관리를 해주는 시스템임.

- 에이전팅 AI 활용 시 업무를 AI에 위임하게 되는데, AI가 잘못된 결과물을 생성하는 경우 이전 버전으로 원복해야 할 상황이 발생할 수 있음.

- 이러한 버전 관리 시스템 활용이 매우 중요함.

- <strong>Git 및 GitHub 연동 설정 방법:</strong>

- 채팅창에 "해당 프로젝트를 Git 초기화해 줘. 그리고 GitHub에도 리모트 퍼블릭 리포지토리를 생성해 줘."라고 요청함.

- <strong>GitHub 설명:</strong> 로컬 Git으로 버전 관리하는 내용을 클라우드 웹상에도 동일하게 싱크하여 관리하는 서비스임.

- 다른 컴퓨터에서도 작업을 이어서 진행하거나, 로컬 파일 문제 발생 시 GitHub에서 받아 작업을 이어갈 수 있는 장점이 있음.

- 따라서 Git 활용 시 GitHub에 문서를 동일하게 싱크하여 버전 관리하는 것을 추천함.

- 위 프롬프트 하나로 코덱스가 Git 초기화 및 GitHub 세팅을 자동으로 진행함.

- 초기 설정 시 GitHub 계정 연동 요청이 있을 수 있으며, GitHub 계정을 생성하고 연동 지침에 따라 진행하면 됨.

- '코덱스 PPT 데모'라는 퍼블릭 저장소를 생성하고 연동하는 과정이 보이며, 완료됨.

## **코덱스(Codex) UI/UX 주요 기능 탐색**

- <strong>채팅 창 기능</strong>

- 채팅창 하단의 '+' 버튼을 통해 포토나 파일 추가 가능함.

- '플랜 모드(Plan Mode)'를 활성화하여 작업을 요청하기 전에 계획을 먼저 받아볼 수 있음.

- '플러그인(Plugin)' 기능을 통해 외부 스킬(외부 툴 연동 기능)을 로드하여 활용할 수 있음.

- <strong>모드(Mode) 설정</strong>

- '모드(Mode)'에서 작업 방식을 설정할 수 있음.

- 처음에는 '오토 리뷰(Auto Review)' 모드를 추천함.

- 이 모드는 코덱스가 웬만한 작업을 진행하다가 민감한 작업에 대해서만 컨펌 요청을 하는 방식임.

- 익숙해지고 세팅이 잘 되었을 때 '풀 엑세스(Full Access)'로 변경하여 활용할 수 있음.

- <strong>작업 환경 설정</strong>

- 'Work Locally': 로컬 폴더에서 작업을 하고 싶을 때 선택함.

- 'Branch': 기본적으로 '메인(Main)'으로 설정함.

- 'Model': 최신 모델인 'GPT-5.5'를 선택함.

- 'Intelligence': '엑스트라 하이(Extra High)'로 설정함.

- <strong>우측 상단 옵션</strong>

- '셋업 런 액션(Setup Run Action)' 기능: 환경 설정 시 필요한 내용을 넣을 수 있으며, 일반적으로는 자주 사용할 일이 없음.

- 우측 끝 버튼: 결과물을 생성했을 때 그 결과물을 확인할 수 있는 창(미리보기)을 띄움. 매우 유용함.

- '토글 터미널(Toggle Terminal)': 터미널 작업을 하단에서 바로 수행할 수 있음.

- 파일 접근 버튼: 파일을 모니터링할 수 있으며, 실제 편집이나 추가 작업을 위해 VS Code 등 다양한 앱으로 접근 가능함.

- <strong>슬래시(Slash) 명령어 활용</strong>

- 채팅 창에 슬래시 입력 시 모델 선택, 퍼스낼리티(Personality) 등 여러 옵션이 나타남.

- <strong>퍼스낼리티 선택:</strong>

- 업무 진행 시 '프래그매틱(Pragmatic)'을 선택하는 것을 추천함.

- 개인적인 대화나 논의 시 '프렌들리(Friendly)'를 선택할 수 있음.

- 사용 가능한 스킬 목록도 확인할 수 있음.

- <strong>펫(Pet) 기능</strong>

- '펫(Pet)'을 활성화하면 화면에 펫이 나타남.

- 펫은 코덱스가 어떤 작업을 진행하고 있는지 알려주며, 코덱스 화면을 보고 있지 않아도 작업 현황을 한눈에 확인할 수 있는 장점이 있음.

## **플러그인 활성화: 프레젠테이션 스킬**

- 왼쪽 상단에 '플러그인(Plugin)' 옵션이 있음.

- 오픈AI가 기본으로 제공하는 다양한 플러그인 목록을 확인할 수 있음.

- 스프레드시트, 프레젠테이션, 슬랙, 노션, JIRA, 구글 드라이브 등이 있음.

- PPT 제작을 위해 '프레젠테이션(Presentation)' 스킬을 활성화함.

- 이를 통해 코덱스가 프레젠테이션 생성을 더 똑똑하게 진행하도록 함.

- 다른 플러그인 활용 예시:

- 컴퓨터 제어를 원할 경우 '컴퓨터 유즈(Computer Use)'를 선택함.

- 브라우저 제어를 원할 경우 '브라우저 유즈(Browser Use)'를 선택함.

## **PPT 제작 전 기본 문서 설정 및 커스터마이징**

- <strong>기본 백그라운드 정보의 중요성</strong>

- 'AI 자동화 컨설팅 제안서 PPT 만들어 줘'와 같이 단순 요청만으로는 원하는 결과물을 얻기 어려울 수 있음.

- 코덱스가 사업, 고객 정보, 말투, 디자인 기준, 검수 기준 등을 전혀 모르는 상황이기 때문임.

- 따라서 작업을 시키기 전에 기본적인 백그라운드 정보를 넣어주는 것이 좋음.

- <strong>GPT-5.5에 최적화된 프롬프트 가이드라인 적용</strong>

- 최근 오픈AI에서 공개한 GPT-5.5에 적합한 프롬프트 가이드를 따르는 것이 좋음.

- 핵심은 과정을 길게 상세히 설명하기보다, 목표, 성공 기준, 제약, 출력, 멈춤 기준 등을 알려주는 것임.

- 즉, 어떻게 할지 전부 지시하기보다, 좋은 결과물이 어떤 모습인지 기준을 명확히 세워주고, 그 기준에 도달하기 위한 방법론은 코덱스가 자유롭게 수행할 수 있도록 구조를 짜 주는 것이 좋은 프롬프트임.

- <strong>초안 문서 생성 요청 및 폴더 구조 설정</strong>

- <strong>목표:</strong> AI 자동화 컨설팅 제안서 PPT 제작용 폴더 준비

- <strong>성공 기준:</strong> 특정 폴더(Agent, Design, Business Context, Output 등) 제작

- <strong>제약 사항:</strong> 아직 PPT 파일 만드는 것은 아님, 초보자가 읽어도 이해되는 쉬운 말로 작성

- <strong>출력:</strong> 특정 파일 형식 (예: .md 파일)

- <strong>멈춤 기준:</strong> 파일 생성 후 다음 작업 진행하지 않고 확인 기다림

- 위와 같은 요청으로 AI 자동화 컨설팅 제안서 PPT 만들기 전 준비 문서 생성을 요청함.

- 폴더 생성뿐만 아니라 그 안에 초안 작성까지 한 번에 진행함.

- <strong>초안 문서 확인</strong>

- 문서 작성이 완료되면 생성된 파일들을 눌러 우측에서 내용을 바로 확인할 수 있음.

- agent.md: 폴더 목적, 작업 원칙, 폴더 구조, 작성 톤 등 포함됨.

- design.md: 디자인 방식에 대한 내용 포함됨.

- business_context.md: 사업 내용에 대한 가안 포함됨.

- deck_brief.md: 슬라이드 덱 생성 시 포함할 내용의 구체적인 구조 설명 문서가 작성됨.

- <strong>문서 구조의 중요성</strong>

- ChatGPT나 Gemini는 프롬프트 하나를 잘 작성하여 특정 결과물을 얻는 식으로 작업하는 경우가 많음.

- 코덱스는 긴 프롬프트 하나보다 기본적으로 코덱스가 참조하여 작업할 수 있는 문서 구조를 잘 작성하는 것이 요청하는 프롬프트보다 훨씬 중요함.

- 따라서 생성된 기본 골격 문서들(agent.md, design.md, business_context.md, deck_brief.md)에 내용을 커스터마이징하는 것이 중요함.

- 원하는 결과물에 대한 의도가 이 네 가지 문서에 정확하게 반영되었는지 체크하고 고도화해 나가면, 코덱스가 그에 맞는 결과물을 잘 생성함.

- <strong>기본 문서 내용 고도화</strong>

- <strong>design.md (디자인 가이드 문서)</strong>

- 초안에는 컬러, 자간, 폰트 등 구체적인 디자인 가이드가 부재함.

- 담당하는 브랜드의 가이드라인이 있다면 그 내용을 넣어주면 좋음.

- 가이드 문서가 없는 경우, getdign.md 같은 사이트에서 유명 브랜드의 design.md 예시를 참고할 수 있음 (예: 스타벅스 design.md).

- 참고 예시 내용을 복사하여 design.md에 붙여넣기 함.

- 이후 "디자인.md 문서에 영문으로 예시를 넣었는데, 한글 제안서에 맞게 한글로 수정해 줘. 그리고 스타벅스 풍은 유지하되 스타벅스 로고나 스타벅스와 관련된 예시는 제거해 줘."와 같이 수정 요청함.

- 코덱스가 좋은 디자인.md 기본 골격을 바탕으로 한글 최적화 및 불필요한 내용 제거 작업을 수행함.

- 수정 결과는 색 사용 기준, 색 조합 예시, 글자 기준, 추천 글꼴, 슬라이드 구성 기준, 자주 쓸 슬라이드 유형 등을 세세하게 정리하여 초안보다 훨씬 자세하고 괜찮게 작성됨.

- 다른 벤치마킹 디자인 파일을 활용하여 우리 브랜드에 맞게 수정 요청하는 방식으로 깔끔한 문서 작성을 유도할 수 있음.

- <strong>business_context.md (비즈니스 컨텍스트 문서)</strong>

- 기존 비즈니스 배경 초안에 몇 가지 인풋을 추가하여 보완 요청함.

- 예시 인풋: 대상 고객(내부 개발팀 없는 10~50명 규모 SMB 대표), 고객의 문제, 제안 서비스, 제공 솔루션 툴(코덱스, 클로드 코드, NA 등), 업무 자동화 솔루션 등.

- "기존 비즈니스컨텍스트.md에 아래 내용을 참고해서 보완해 줘."와 같이 요청함.

- 코덱스가 추가 정보를 활용하여 기존 초안의 의미 있는 내용과 함께 보완 작업을 수행함.

- 결과적으로 예상 고객, 제안 서비스, 제공 솔루션 툴 등이 훨씬 구체화됨.

- <strong>agent.md (작업 지침 문서)</strong>

- 동일하게 인풋을 주면서 agent.md 내용 보완을 요청함.

- 프롬프트 작성 시 목표, 제작 규칙, 안전 규칙 등 결과물이 어떤 식으로 나와야 하는지 기준으로 설명하는 것이 중요함.

- 특히 agent.md는 코덱스가 작업 지침을 수행하는 기본 문서이므로, 원하는 목표에 대해 구체적으로 이해하고 작업을 할 때 지켜야 할 원칙 등을 모두 확인할 수 있게 해주는 것이 중요함.

- 생성한 별도의 문서들(design.md, business_context.md, deck_brief.md)을 추가 작업 시 참조할 수 있도록 링크를 걸어주는 것이 중요함.

- 예: 브랜드 디자인 작업 시 design.md 참조, 고객 문제 파악 및 사업 내용 파악 시 business_context.md 참조, 슬라이드 흐름 확인 시 deck_brief.md 참조.

- <strong>deck_brief.md (슬라이드 구성 문서)</strong>

- 어느 정도 구체적으로 적혀 있지만, 원하는 방향과 다른 부분이 있을 수 있으므로 보완 요청함.

- 이때 비즈니스 컨텍스트 문서도 수정되었으므로, 그 맥락에 맞춰 수정해 달라고 요청할 수 있음.

- 코덱스가 프로젝트 폴더 내 파일들을 참조하여 작업을 진행하는 것을 확인할 수 있음.

- 수정 결과 주 슬라이드 흐름, 각 슬라이드에서 말할 내용, 필요한 자료, 정해야 할 것들 등이 포함되며, 지정한 툴(코덱스, 클로드 코드, NA)도 반영됨.

- 최근 ChatGPT의 새로운 이미지 모델을 코덱스 안에서도 활용할 수 있으므로, PPT에 들어갈 이미지 추가 내용을 deck_brief.md에 넣어주는 것이 좋음.

- "덱 브리프랑 에이전트.md에 이미지가 필요한 영역은 이미지 젠(Image Gen)을 이용해서 삽입할 수 있게 수정해 줘. 그리고 전체 덱에서 이미지가 최소 세 개 정도 활용되는 것을 권장해 줘."와 같이 요청함.

- 이미지 생성과 관련하여 덱에 어떤 식으로 접목시킬지 내용이 추가됨.

## **제안서 PPT 제작 요청 및 계획 수립**

- 네 가지 문서(agent.md, design.md, business_context.md, deck_brief.md) 세팅이 완료됨.

- <strong>'플랜 모드(Plan Mode)' 활용</strong>

- 바로 최종본을 요청하기보다, 먼저 코덱스가 어떻게 접근할지 계획을 수립하도록 요청하고, 필요시 수정 요청을 하는 것이 좋음.

- PPT 덱 제작은 무거운 작업이므로, 사용량을 효율적으로 관리하기 위해 계획을 최적화한 후 원하는 결과물을 한 번에 생성하는 것이 효율적임.

- 채팅창 하단의 '+' 버튼을 눌러 플랜 모드를 켜고 요청함.

- <strong>계획 수립 요청 예시</strong>

- "여성용 가방을 판매하는 온라인 쇼핑몰 대상 열 장짜리 AI 자동화 컨설팅 제안서 PPT를 만들고 싶다."와 같이 요청함.

- 성공 기준에 대해 잘 설명하고, 해당 업계의 페인포인트나 적절한 솔루션에 대한 리서치도 함께 요청함.

- <strong>계획 수립 과정 및 질문 응답</strong>

- 코덱스가 '계획 수립 단계'임을 파악하고 PPT를 만들지 않고 먼저 계획을 수립함.

- 코덱스가 리서치 근거 시장 기준을 질문할 수 있음 (예: "리서치 근거는 어느 시장 기준으로 잡을까요?"). 한국 중심으로 하겠다고 응답함.

- 최종 목적에 대한 질문에 (예: "최종 목적이 무엇이냐?") 파일럿 미팅이라고 응답함.

- 고객사 표현 방식에 대한 질문에 (예: "고객사는 어떻게 표현할까요?") 실명 반영을 하겠다고 응답함.

- <strong>계획 검토 및 확정</strong>

- 코덱스가 목표 요약, 제안 목적, 시장 기준, 디자인 및 슬라이드 구성안, 핵심 메시지 등을 정리한 계획을 보여줌.

- 깔끔하게 정리된 계획을 '익스팬드 플랜(Expand Plan)'을 통해 확인 가능함.

- 여기서 수정하고 싶은 부분이 있다면 먼저 수정을 요청하여 토큰 낭비 없이 방향을 한 번에 확정할 수 있음.

- 현재는 괜찮다고 가정하고 진행함.

- <strong>고객사 정보 입력 및 작업 시작</strong>

- 계획 승인 후, 고객사 명을 알려달라는 요청이 있을 수 있음.

- 예시로 '그린 토트백'이라는 가상의 고객사 명을 입력함.

- 로고 제공 여부를 질문할 수 있으며, 현재는 무로고로 진행함.

- 이후 코덱스가 웹 리서치부터 진행하는 것을 확인할 수 있음.

## **제안서 PPT 최종 제작 및 검토**

- <strong>PPT 생성 완료 및 소요 시간</strong>

- 약 20분 정도 소요되어 슬라이드 생성이 완료됨.

- 리서치, 이미지 생성, PPT 제작, 그리고 검수까지 한 번에 진행함.

- <strong>생성된 추가 문서 확인</strong>

- 검수한 내용도 요약되어 있음.

- 리서치한 문서(온라인몰 리서치 요약, AI 자동화 솔루션 맵)도 생성되어 있음.

- 별도로 작성된 슬라이드 계획을 바탕으로 PPT 제작이 완료됨.

- <strong>생성된 PPT 내용 확인</strong>

- PPT 파일을 누르면 바로 간단하게 리뷰할 수 있는 창이 띄워짐.

- design.md 문서에 스타벅스 풍으로 디자인 가이드를 작성했으므로, 그 느낌이 반영된 것을 확인할 수 있음.

- 고객사를 '그린 토트백'이라는 가상의 업체로 선정했으므로, 그에 맞게 이미지도 잘 제작하여 삽입된 것을 확인할 수 있음.

- 슬라이드 내용과 이미지가 적절하게 배치되어 있음.

- 이미지 3개 삽입 요청에 따라 슬라이드 1번, 3번, 6번에 이미지가 추가됨.

- 특화 문제 제시, 자동화 업무 3가지 선택, 코덱스를 업무 자동화 설계 허브로 둔 활용 방안, 4주 진행 계획, 수치로 확인하는 효과, 운영 리스크 관리, 다음 미팅에서 파일럿 범위만 정하면 된다는 최종 메시지 등 원샷으로 괜찮은 수준의 제안서가 작성됨.

- <strong>제작의 성공 요인</strong>

- 리서치 및 수치들이 실제 리서치 기반으로 뽑아진 것을 알 수 있음.

- 이처럼 잘 작동하게 하기 위해서는 앞서 설정했던 기준 문서를 잘 세팅하고 계획을 잘 수립하는 것이 매우 중요함.

- 어설픈 문서와 계획 또는 계획 없이 PPT 작업을 요청하면 수십 분에 걸쳐 작업했더라도 의미 없는 결과물이 나올 수 있음.

## **PPT 추가 검수 및 수정**

- <strong>플러그인을 활용한 추가 검수 요청</strong>

- 생성된 PPT가 괜찮지만, 혹시 모를 상황에 대비하여 추가 검수를 요청할 수 있음.

- 채팅창에 '@'를 입력하여 플러그인을 호출하고, '프레젠테이션(Presentation)'을 선택함.

- "방금 완료된 ppt를 한 번 더 검수해 줘."라고 요청하고, 레이아웃 측면과 내용 측면에서 개선 포인트를 요청할 수 있음.

- 실제로 제안서 작성 시 내용이나 흐름 측면에서 의견이 있을 수 있으므로, 이 경우 피드백과 함께 성공 기준을 작성하여 코덱스가 기준에 맞게 개선하도록 유도함.

- <strong>리뷰 몬타주 PNG 파일 생성</strong>

- 슬라이드 형태로도 제공되지만, 한눈에 덱의 구성 느낌을 볼 수 있도록 '리뷰 몬타주 PNG' 파일도 함께 생성됨.

- 이를 통해 흐름에 맞춰 슬라이드가 어떻게 진행되는지 한눈에 파악할 수 있어 유용함.

- <strong>검수 결과 확인</strong>

- 검수가 진행된 후 결과가 나옴.

- 렌더링 확인 문제는 없었고, 팩트 체크 및 추가 확인이 필요한 몇 가지 사항이 지적됨.

- <strong>검수 결과 반영 및 내용 수정</strong>

- 제안된 내용처럼 슬라이드 2번과 6번에 대해 정보 관련 팩트 체크 및 필요시 수정 요청함.

- 특히 수치나 툴 관련 내용이 과장되거나 틀린 정보는 없는지 더블 체크가 필요함.

- 항상 소스 문서를 참조해 달라고 요청하면, 코덱스가 소스 문서를 들어가 내용을 추가적으로 확인하여 수정함.

- 슬라이드 2번과 6번의 내용이 최신 정보 및 통계 기반으로 수정됨.

## **최종 PPT 파일 실행 검수 (컴퓨터 유즈 기능 활용)**

- <strong>파워포인트 실행 환경 검수 필요성</strong>

- 슬라이드 작업 시 분명히 만든 환경에서는 잘 나오더라도, 실제 파워포인트에서 실행 시 깨지거나 어색하게 보이는 경우(줄 바꿈, 폰트 깨짐 등)가 발생할 수 있음.

- 이러한 내용도 체크할 필요가 있음.

- <strong>컴퓨터 유즈(Computer Use) 활성화 및 검수 요청</strong>

- 코덱스에 '컴퓨터 유즈' 기능을 활용하여 해당 작업마저 요청할 수 있음.

- '@'를 입력하여 '컴퓨터(Computer)' 기능을 활성화하고, "파워포인트를 실행해서 최종적으로 ppt를 열고 확인해 줘."라고 요청함.

- 코덱스가 파워포인트를 실행하여 최종 결과물이 괜찮은지 직접 체크하고 결과 리포트를 생성함.

- 이로써 제안서 작성의 리서치부터 최종 파워포인트 실행 검수까지 엔드 투 엔드(End-to-End)로 코덱스가 모든 과정을 처리할 수 있게 됨.

- <strong>자동화된 검수 과정</strong>

- 코덱스의 '컴퓨터 유즈'가 직접 파워포인트를 넘기면서 장표별로 확인하는 과정을 거침. (화면에 보이는 마우스 포인터는 컴퓨터 유즈의 포인터임.)

- 편집 화면 기준으로 1번부터 10번까지 모두 정상적으로 열렸고 문제가 없었음을 확인함.

- 이후 슬라이드 쇼를 처음부터 실행하여 10장을 전체 넘김으로 다시 확인하는 과정을 거침.

- 사람이 편집 화면과 슬라이드 쇼로 보는 것과 동일한 방식으로 코덱스도 작업을 진행함.

- 슬라이드 1번부터 10번까지 모두 정상 표기되었고, 특별한 문제가 보이지 않았음을 확인함.

- <strong>최종 보고서 정리</strong>

- 최종 파일과 검수 결과를 정리하고, 다음에 템플릿화하면 좋은 정보가 있을지 의견을 요청함.

- 코덱스가 산출물 목록, 검수 결과 요약, 사람이 마지막으로 확인해야 할 사항, 템플릿화 제안 등의 내용을 담은 최종 정리 문서를 생성함.

- 여기서 추가적으로 템플릿화하고 싶은 내용이 있다면 문서를 고도화하여, 다음에는 고객사 정보만 주어도 제안서를 원하는 스타일로 생성할 수 있도록 함.

## **코덱스(Codex) 효율적 활용을 위한 팁**

- <strong>메모리(Memory) 활용</strong>

- 코덱스가 이전에 요청했던 내용을 기억하고 작업을 수행하도록 하려면 '슬래시(/) 메모리스(Memories)'에 들어가 '메모리 사용' 및 '메모리 생성'을 활성화해야 함.

- <strong>반복 작업 시 스킬(Skill) 생성 및 활용</strong>

- 오픈AI가 제공하는 플러그인 외에, 내 워크플로우에서 의미 있는 반복 작업이 있다면 스킬을 생성하여 활용하는 것이 좋음.

- '@ 스킬 크리에이터(Skill Creator)'를 사용하여 스킬을 만들고, 해당 스킬로 작업을 진행하면 반복 작업을 일관되게 실행할 수 있음.

- 다른 사람이 GitHub에 올려놓은 스킬이 있다면 '@ 스킬 인스톨러(Skill Installer)'를 사용하여 URL을 넣고 설치 요청하면 자동으로 설치됨.

- 항상 스킬을 활용하여 반복 작업을 진행하는 것을 추천함.

- <strong>훅(Hook) 기능 활용</strong>

- agent.md 문서만으로 GPT-5.5가 웬만한 작업을 잘 실행하지만, 작업이 복잡해질수록 놓치는 경우가 발생할 수 있음.

- 모델이 특정 부분을 놓칠 때, '훅(Hook)'을 활용하여 무조건 실행하게 할 수 있음. (일종의 코드 작업으로 자동화를 하는 것과 유사함.)

- 예: 결과물 생성 후 특정 체크리스트를 항상 확인하고 검증하는 작업을 요청할 때, agent.md에 내용이 있음에도 제대로 실행되지 않는 경우 '훅'을 활용함.

- 코덱스와 대화하여 훅 세팅 방법을 안내받고 작업에 적용할 수 있음.

- 일반적인 간단한 작업은 agent.md 최적화만으로도 충분하지만, 놓치는 부분이 생긴다면 '훅'을 고려하는 것이 좋음.

## **코덱스(Codex) 활용의 핵심 요약**

- <strong>1. 코덱스 앱은 채팅방이 아니라 프로젝트 폴더 기반 앱임.</strong>

- 프로젝트 폴더를 잘 생성하고 Git 세팅을 하는 것이 중요함.

- <strong>2. 좋은 결과물은 긴 프롬프트 하나보다 좋은 작업 환경(문서 구조)에서 나옴.</strong>

- 에이전트.md, 디자인.md 등 핵심 문서들을 잘 세팅하는 것이 매우 중요함.

- <strong>3. AI 결과물은 생성보다 계획 수립과 검수가 매우 중요함.</strong>

- 생성 요청 전에 '플랜(Plan)'을 받아보고 고도화하는 것을 추천함.

- 플랜 모드에서 계획을 잘 수립한 후 생성을 진행하고, 생성된 결과물에 대해 검수 작업을 요청할 수 있음.

- 한 번 잘 진행된 작업은 메모리, 스킬, 훅 같은 기능을 활용하여 반복 작업을 효율적으로 진행하는 것이 좋음.
